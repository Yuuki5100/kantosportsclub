## 2. チーム全員で同じ環境を使うための推奨構成

### 2.1 Dev Containers（最優先）

WSL + Docker が既にあるため、`.devcontainer/devcontainer.json` を作成するだけで VSCode で全員同一環境を再現できる。

```
.devcontainer/
  devcontainer.json   ← Node.js・Java・PlantUML 等のバージョン固定
  docker-compose.yml  ← PlantUML サーバー + 開発コンテナを一括起動
```

**メリット：**
- 新メンバーが VSCode でリポジトリを開く → "コンテナで開く" → 即開発開始
- NVM の手動セットアップ・依存ライブラリのインストールが不要になる

### 2.2 mise（NVM の代替・チーム向け）

NVM はシェル設定への追記が必要でチームでの統一が難しい。
[mise](https://mise.jdx.dev/) はプロジェクトルートの `.mise.toml` で Node / Java / Python のバージョンを宣言でき、`mise install` 1 コマンドで揃う。

```toml
# .mise.toml（プロジェクトルートに置く）
[tools]
node = "24.14.0"
java = "openjdk-21"
```

## 3. テスト自動化で採用を推奨する方式（GitLab Self-Managed / GitLab Free 前提）

### 3.1 CI/CD（必須）

| 方式 | 推奨理由 | 前提・制約 |
|------|---------|------------|
| **GitLab CI/CD（`.gitlab-ci.yml` + `CI/ci-config/*.yml`）** | 既存パイプライン資産をそのまま拡張でき、MR / branch pipeline に統合しやすい | Self-Managed / Free 前提。Runner の `privileged` / `docker.sock` / `pull_policy` / `concurrent` は管理設定依存のため要確認 |

**当面のパイプライン方針（MR / develop / release 系）：**

```
validate → backend test/coverage → backend build → frontend lint/test/coverage（段階導入） → E2E / k6 / ZAP（Runner 条件確認後に手動・定期実行へ拡張）
```

### 3.2 ビジュアルリグレッションテスト

| 方式 | 推奨理由 | 前提・制約 |
|------|---------|------------|
| **Storybook build + artifacts 保存** | GitLab Free でも利用でき、まずは UI カタログ生成物を CI 成果物として確認できる | MR 上の高度な UI 差分レビューは GitLab ネイティブでは不足するため、必要なら Chromatic / reg-suit 等の外部ツール併用を別途検討 |

Storybook 8 が既に導入済みのため連携コストが低い。

### 3.3 カバレッジ可視化

| 方式 | 推奨理由 | 前提・制約 |
|------|---------|------------|
| **JaCoCo / Jest coverage を GitLab artifacts へ保存** | GitLab Free でも実現でき、まずは HTML / XML レポートをジョブ成果物で確認できる | artifact のデフォルト保持期限は 10 days、最大サイズは 4096MB。長期保管が必要な場合は別保管先の設計が必要 |
| **Cobertura / coverage report 連携** | MR 上で最低限の coverage 把握に使える可能性がある | 現行 job 定義と GitLab バージョンでの表示範囲は要確認。Codecov 前提にはしない |

`jest --coverage` と `jacoco:report` の成果物をまず artifact 化し、その後に MR 表示を追加する順で進める。

### 3.4 コード品質・静的解析

| 方式 | 推奨理由 | 前提・制約 |
|------|---------|------------|
| **ESLint / TypeScript / OWASP Dependency-Check / SpotBugs 等の CLI レポートを artifacts 保存** | GitLab Free でも実行しやすく、まずは CI ログと成果物で品質確認できる | GitLab Code Quality UI への強い依存は避ける。品質ゲートはジョブ失敗条件として段階的に導入する |
| **GitLab SAST** | 実行自体は選択肢になる | Free 前提では MR 上の高度な差分表示・UI 活用が限定的なため、レポート出力中心の補助扱いにする |

許可情報を扱うシステムのため、セキュリティ観点での静的解析は特に有用。

### 3.5 負荷テスト

| サービス | 推奨理由 | コスト |
|---------|---------|--------|
| **k6** | JavaScript / TypeScript でシナリオ記述・Docker 実行可・SA03 非機能要件（同時接続数・応答時間）に対応した検証が可能 | OSS 無料 |

ST フェーズ（2026/11〜）に向けて基本設計中にシナリオ定義だけ進めておくと効率的。

### 3.6 セキュリティテスト

| 方式 | 推奨理由 | 前提・制約 |
|------|---------|------------|
| **OWASP ZAP（Docker イメージ）を GitLab CI から手動・定期実行** | GitLab DAST が使えない前提でも、ZAP レポートを artifact として残せる | `docker compose` / DinD / `docker.sock` 利用可否は Runner 設定に依存するため要確認。本番向け常時スキャンではなく検証環境向けに限定する |
| **GitLab DAST** | - | GitLab Free 前提のため採用しない |

## 5. GitLab Self-Managed / GitLab Free 前提の実装計画（修正版）

### 5.1 現在の GitLab 制約一覧

#### 5.1.1 確定している制約

| 区分 | 内容 | 設計への影響 |
|------|------|--------------|
| GitLab 環境 | GitLab は self-hosted、かつ有効なサブスクリプションなし（実質 GitLab Free） | GitLab Ultimate 前提の MR セキュリティ UI / DAST / 高度な Code Quality 表示を前提にしない |
| Runner | Docker executor の Runner は存在する | CI は Docker image ベースの job を前提にできるが、Compose / DinD 利用可否は別途確認が必要 |
| artifacts | CI/CD 全体設定で artifact 最大サイズは 4096MB、デフォルト保持期限は 10 days | テストレポートは原則 artifact 保存でよいが、長期保持が必要な成果物は別保管先または手動退避方針が必要 |
| Runner 作成権限 | project / group メンバーが Runner を作成できる | 専用タグ付き Runner を追加できる可能性はあるが、設定値は管理ルール確認が必要 |
| GitLab ネイティブ機能 | DAST は使えない。SAST は実行できても MR 上の高度な差分表示が弱い。Code Quality の GitLab UI 活用も限定的 | 品質・セキュリティ検証は GitLab UI 依存ではなく、CLI 実行 + job failure + artifacts レポート中心で設計する |
| 現行 CI 構成 | stages は `validate → test → coverage → build → migrate-db → deploy`。workflow は `release_dev` / `release_uat` 向け deploy 制御中心。`BACKEND_ONLY=true` が既定 | 現行 pipeline は backend 偏重であり、frontend / Storybook / Playwright / k6 / ZAP の常時実行を一気に入れる前提にしない |
| 現行 include / tags | `CI/ci-config/common.yml` / `backend.yml` / `frontend.yml` を include。既存 job は `docker-runner` タグを使用 | 新規 job も Runner tags を明示し、既存 include 分割を崩さず段階追加する方針が現実的 |

#### 5.1.2 要確認事項

| 項目 | 確認内容 | 未確認のまま進めた場合のリスク |
|------|----------|------------------------------|
| Runner privileged | 対象 Runner で `privileged=true` が許可されているか | `docker:dind` や一部の Docker ベース E2E / ZAP 実行が失敗する可能性がある |
| docker.sock 利用 | Runner コンテナからホスト `docker.sock` を利用できるか | `docker compose` や `docker/stack.sh` を CI から直接呼べるか判断できない |
| pull_policy | 社内 Registry / 外部 Registry からの image pull 方針、`if-not-present` 等の設定可否 | 外部 image 取得失敗、想定外の最新版 pull、ビルド再現性低下が起きる可能性がある |
| concurrent | Runner の並列実行数、同一タグ Runner の実効キャパシティ | Playwright / k6 / ZAP を追加した際に queue 滞留や CI 時間悪化が起きる可能性がある |
| Runner tags 標準 | `docker-runner` 以外の用途別タグを作るか、E2E / 非機能専用 Runner を分けるか | 重い job が通常 CI を圧迫し、MR 開発の待ち時間が増える可能性がある |
| Docker Compose 実行方式 | `docker compose` を Runner 上で使うか、service container / テスト専用 job image に寄せるか | `docker/stack.sh` をそのまま CI 化できるか、別実装が必要かが確定しない |
| GitLab バージョン差 | Cobertura / JUnit / coverage 表示、`artifacts:reports` の利用可能範囲 | レポートを artifact 化できても MR 画面で期待どおり見えない可能性がある |
| 長期レポート保管 | 10 days を超えて保持したい k6 / ZAP / coverage レポートの保管先と保存期間 | ST 証跡や監査用レポートが artifact expiration により消える可能性がある |
| 外部 SaaS 利用可否 | Chromatic / Codecov / SonarCloud 等の外部サービス利用可否、ネットワーク制限、契約可否 | 外部サービス前提の UI 回帰・可視化計画が社内制約で採用できない可能性がある |

#### 5.1.3 制約ではない事項

| 項目 | 補足 |
|------|------|
| Runner の `executor` / `privileged` / `pull_policy` / `concurrent` | これらは GitLab Free のプラン制約ではなく、self-managed の管理設定依存。ただし現状値は要確認 |
| artifact 利用そのもの | GitLab Free でも job artifacts は利用可能。今回の制約は最大サイズ 4096MB と保持期限 10 days |
| Runner tags の明示 | Free / 有償プラン差ではなく CI 設計上の運用ルール。むしろ「どの Runner でも動く」前提を避けるため明示すべき事項 |
| Frontend / E2E / k6 / ZAP の導入可否 | プラン上ただちに不可というより、Runner 設定・実行方式・保管方針が固まれば段階導入できる領域 |

### 5.2 実装計画の問題点

元の計画のうち、現在の GitLab 環境とズレている点は以下のとおり。

| 問題点 | ズレの内容 | 修正方針 |
|--------|------------|----------|
| GitHub Actions 前提 | CI 基盤が GitLab なのに `.github/workflows/ci.yml` 追加を前提としていた | `.gitlab-ci.yml` と `CI/ci-config/*.yml` の拡張計画へ置き換える |
| Codecov / SonarCloud 前提 | GitLab Free / self-managed での現実運用より外部 SaaS 前提が強い | まず artifact + CLI レポート中心にし、外部 SaaS は要確認事項として後段検討に下げる |
| GitLab DAST 前提に近い記載 | GitLab DAST は Free 前提では使えない | ZAP を独立 job として artifacts 出力する方針に限定し、GitLab DAST は採用しない |
| GitLab UI 依存が強い | SAST / Code Quality の MR 差分表示を前提にした品質ゲート設計になっていた | Free でも成立する `script` の閾値判定、job failure、JUnit/HTML/XML artifact を中心にする |
| `docker/stack.sh` を CI から直接呼ぶ前提 | `privileged` / `docker.sock` / DinD / `docker compose` 可否が未確認 | Runner 要件を要確認として明示し、確認前は Maven / Node image で完結する job を優先する |
| 全テストの一括常時実行前提 | 現行 `.gitlab-ci.yml` は `BACKEND_ONLY=true` 既定で backend 偏重。frontend / E2E / k6 / ZAP を一気に常時実行する設計と合っていない | backend test / coverage → frontend lint/test → E2E → 非機能テストの順で feature flag 付き段階導入にする |
| artifact 長期保持が未考慮 | 既存 artifact は 1h / 24h が多く、GitLab デフォルトも 10 days | MR 調査用と ST 証跡用を分け、長期保管が必要な成果物は別保管先または release 時手動退避を明記する |
| Runner 標準化の前提が不足 | どの Runner でも動く前提で書かれていた | `tags`、必要 image、Docker 利用条件、想定 Runner 種別を成果物・完了条件に含める |

### 5.3 修正版の実装計画

| フェーズ | 目的 | 実施内容 | GitLab 側の前提条件 | 成果物 | 完了条件 |
|---------|------|----------|--------------------|--------|----------|
| Phase 1: 既存 backend CI の安定化 | 現行 backend 偏重 pipeline を前提に、まず壊れにくい自動テスト基盤を固める | `CI/ci-config/backend.yml` の `backend-test` / `backend-coverage` / `backend-dependency-scan` をレビューし、JUnit / JaCoCo / Dependency-Check レポートを artifacts として整理する。`BACKEND_COVERAGE` / `BACKEND_ITEST` の運用条件も明文化する | 既存 `docker-runner` タグの Docker executor Runner が利用可能。Maven image を pull できること。Compose / DinD はこの段階では前提にしない | 更新済み `CI/ci-config/backend.yml`、backend coverage / test / dependency scan artifacts、運用手順 | MR または `develop` pipeline で backend test が安定実行され、失敗時に surefire / jacoco / dependency-check 成果物を確認できる |
| Phase 2: Frontend lint / Jest / coverage の段階導入 | `BACKEND_ONLY=true` 既定を維持しつつ、frontend 検証をフラグ制御で安全に追加する | `CI/ci-config/frontend.yml` の `frontend-lint` / `frontend-jest-test` / `frontend-coverage` を整備し、`FRONTEND_LINT` / `FRONTEND_TEST` / `FRONTEND_COVERAGE` を MR 単位で有効化できるようにする。Storybook はまず build 成功確認と artifact 保存に留める | Node image を pull できること。`docker-runner` タグまたは frontend 用タグが利用可能。node_modules cache 容量と実行時間を要確認 | 更新済み `CI/ci-config/frontend.yml`、Jest / coverage / Storybook build artifacts、frontend 有効化手順 | backend 既存 CI を壊さず、必要な MR で frontend lint / test / coverage を有効化でき、成果物を artifacts で確認できる |

| Phase 4: Runner 条件確認と CI 実行基盤の標準化 | E2E / k6 / ZAP を入れる前に、Runner 依存条件を確定する | Runner の `config.toml`、`privileged`、`pull_policy`、`concurrent`、`docker.sock` マウント有無、利用可能タグ、Docker / Compose バージョンを確認する。E2E / 非機能専用 Runner タグを分けるかも決める | self-managed 管理者設定の確認が必要。project/group メンバーが Runner を作成できるが、利用ルールは要確認 | Runner 要件一覧、推奨 tags 設計、`docker/stack.sh` を CI から呼ぶ可否判断、必要なら代替実行方式の設計メモ | 「Compose / DinD を使う job」「使わない job」の境界が明確になり、どの job がどの Runner tag を要求するか確定している |
| Phase 5: Playwright E2E の限定導入 | 最初から常時 E2E にせず、失敗調査可能な形で限定運用を始める | Runner 条件確認後、Playwright smoke を手動 job または schedule job で追加する。`docker/stack.sh e2e` 直呼びが難しい場合は、service container や専用 job image で最小 smoke を再実装する。HTML report / trace / screenshot を artifacts 保存する | `docker compose` / DinD / `docker.sock` 利用可否は要確認。利用不可なら stack.sh 直呼び前提を避ける。Playwright 実行用 Runner tag を明示する | E2E job 定義、Playwright report artifacts、E2E 実行手順、必要なら seed 手順 | 限定条件の pipeline で smoke E2E が再現性をもって実行でき、失敗時に trace / report を artifact から確認できる |

| Phase 7: テストデータ統一 | backend / frontend / E2E / 非機能テストの前提データを揃え、環境差による揺れを減らす | `docker/mysql/initdb.d`、テスト fixture、E2E seed、k6/ZAP 向けアカウント・権限・対象 URL を整理する。CI 用 seed とローカル Docker 用 seed の差分を最小化する | DB 初期化方式、テスト環境 URL、認証情報の GitLab CI Variables 管理ルールを要確認 | seed SQL / seed script、テストデータ仕様、CI Variables 管理手順 | ローカルと CI で同じ前提データを再現でき、E2E / k6 / ZAP の失敗原因を環境差から切り分けやすくなる |

### 5.4 実装計画の修正後サマリ

- **この順序にした理由**  
  現行 `.gitlab-ci.yml` が `BACKEND_ONLY=true` 既定で backend 偏重であるため、まず既存 backend job の信頼性とレポート整備を優先し、その後 frontend、最後に Runner 条件依存が強い E2E / k6 / ZAP へ広げる順序が最もリスクが低い。

- **GitLab Free / self-managed での現実解として優先したこと**  
  GitLab Ultimate 前提の MR セキュリティ UI / DAST / 高度な Code Quality 表示ではなく、Free でも確実に使える `script` 判定、job failure、JUnit/JaCoCo/Jest/ZAP/k6 の artifacts 保存、Runner tags 明示、手動/定期 job による段階導入を優先した。

- **長期保管の考え方**  
  MR 調査用レポートは GitLab artifacts の 10 days 前後で十分とし、ST 証跡・監査向けに 10 days を超えて残す必要がある k6 / ZAP / coverage レポートは、別ストレージ退避または release 時の手動保存を設計対象として切り出す。
