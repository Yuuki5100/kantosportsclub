
```markdown# 🏗️ 双方向DB同期機能の実装ステップ一覧

## 🧩 構成概要

- 共通モジュール群（通信、署名、再送、ログなど）を先に実装
- その後、操業管理アプリ・入出庫管理アプリでこの共通機能を用いた個別同期処理を実装

---

## ✅ 段階①：共通機能 - サーバー間通信モジュール群の実装

### 📁 目的
双方向DB同期を支える、サーバー間通信の**共通インフラモジュール**をライブラリ化・整備する。

### 🔧 作業一覧

| No | 内容 |
|----|------|
| 1-1 | `SyncRequest` / `SyncResult` など基本DTOの設計 |
| 1-2 | `SyncRequestSender` インターフェース定義（送信汎用） |
| 1-3 | `HttpSyncRequestSender` 実装（RestTemplate/WebClient利用） |
| 1-4 | `HmacSignatureUtil` 実装（HMAC署名・検証処理） |
| 1-5 | `RetryableSyncManager` 実装（再送キュー登録・リトライ処理） |
| 1-6 | `PendingSyncRepository` のインターフェース定義（JPA/Redis/DBなど） |
| 1-7 | `SyncFailureNotificationService` の共通インターフェース定義（Slack, Teams対応） |
| 1-8 | `SyncRequestLogger` 実装（requestId含むロギング） |
| 1-9 | 上記モジュール群のユニットテスト整備 |
| 1-10 | 共通基盤（例：`servercommon` モジュール）に組み込み・依存性登録 |

---

## ✅ 段階②：①を用いた個別アプリでの同期処理実装

### 📁 目的
操業管理システム → 入出庫管理システムへの同期処理を、共通機能を使って実装。

### 🔧 作業一覧（操業管理アプリ側）

| No | 内容 |
|----|------|
| 2-1 | `/api/update-shared-table` エンドポイントの実装 |
| 2-2 | 操業DBの更新処理（トランザクション対応） |
| 2-3 | `SyncRequest` を構成し、`SyncRequestSender.send()` で入出庫API呼び出し |
| 2-4 | 入出庫側API呼び出し失敗時の再送キュー登録 |
| 2-5 | `requestId` 発行・付与（冪等性対応） |
| 2-6 | `SyncRequestLogger` による操作ログ出力 |
| 2-7 | 再送バッチ（`@Scheduled` or ジョブ）の設定と有効化 |
| 2-8 | 結果の整形とHTTPレスポンス返却 |
| 2-9 | 統合テスト（DB更新 + API呼び出し + 再送）

---

### 🔧 作業一覧（入出庫管理アプリ側）

| No | 内容 |
|----|------|
| 2-10 | `/api/sync-shared-table` の受信API実装 |
| 2-11 | HMAC署名の検証処理 |
| 2-12 | `requestId` による冪等性制御（受信済み判定） |
| 2-13 | 入出庫DBの更新処理（トランザクション対応） |
| 2-14 | ロギング・操作ID出力 |
| 2-15 | 入出庫側のユニット/統合テスト

---

## 📝 備考

- 各アプリでの環境ごとの `sync.api.url`, `sync.secret.key` 等は application.yml で管理
- 本構成は他アプリでも再利用できるよう、共通モジュールを疎結合に保つ
- 再送対象の同期処理単位は、requestIdで一意管理されるべき
```
