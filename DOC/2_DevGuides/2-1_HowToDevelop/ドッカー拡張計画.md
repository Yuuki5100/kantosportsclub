# ドッカー拡張計画（nvm / Newman / Python）

## 1. 結論

- 現状、Dev Container には `nvm` が未導入。
- Node.js は `.devcontainer/Dockerfile` で公式バイナリを直接展開して導入している。

## 1.1 正式決定（2026-04-09）

- `nvm` は不採用とする（理由: Node.js 管理の二重化を避けるため）。
- Node.js は現行どおり `.devcontainer/Dockerfile` の固定バージョン管理を継続する。
- タイムゾーン統一はコンテナ実行環境（`TZ=Asia/Tokyo`）を対象とし、アプリの DB 接続設定（`connectionTimeZone=UTC` / `hibernate.jdbc.time_zone=UTC`）は変更しない。

## 2. 現状根拠

- `.devcontainer/Dockerfile` に `nvm` のインストール処理がない
- `.devcontainer/post-create.sh` に `nvm` 初期化処理がない
- `.devcontainer/devcontainer.json` に `nvm` 利用前提の設定がない

## 3. 拡張方針

- 方針A（採用）: 現行の Node.js 直接導入を維持し、`nvm` は導入しない
- 方針B: 開発者操作用に `nvm` を追加し、デフォルト Node.js は `NODE_VERSION` と一致させる

推奨理由（方針A）:
- 現在すでに Node.js 24.14.0 で固定化できており、再現性が高い
- `nvm` はシェル初期化や運用手順の追加が必要で、設定差異が入りやすい

## 4. 方針Bを採用する場合の実装計画

1. `.devcontainer/Dockerfile` に `nvm` インストール処理を追加  
2. `vscode` ユーザーのシェル初期化ファイルへ `NVM_DIR` と `nvm.sh` 読み込みを追加  
3. `nvm install ${NODE_VERSION}` と `nvm alias default ${NODE_VERSION}` を実行  
4. `node` / `npm` が非対話シェルでも参照できるよう PATH を固定  
5. `04_devcontainer_architecture.md` の 4.1 に `nvm` を追記  
6. 動作確認: `nvm --version`, `node -v`, `npm -v`

## 5. 影響範囲

- `.devcontainer/Dockerfile`
- 必要に応じて `.devcontainer/post-create.sh`
- `DOC/2_DevGuides/2-1_HowToDevelop/04_devcontainer_architecture.md`

## 6. リスク

- シェル初期化依存で `nvm` が読まれない場合、`node` 解決が不安定化する
- 既存の Node.js 直接導入と二重管理になると保守性が下がる

## 7. 受け入れ条件

- `Rebuild Container` 後、`nvm --version` が成功
- `node -v` が `v24.14.0` を返す
- `npm ci` と frontend テストが従来どおり実行可能

## 11. 開発系統マルチ化（3系統＋拡張）改修計画

## 11.1 目的

- 既存リポジトリ（モダン構成）を土台に、異なる技術系統へ Dev Container / Docker 運用を横展開できる共通枠組みを作る。
- 初期対象は以下 3 系統とし、4つ目以降を追加しやすい構造を標準化する。
  - 1) モダン環境（現行リポジトリ）
  - 2) C# 系列
  - 3) Java レガシー（Apache + Tomcat など）
  - 4) 随時拡張

## 11.2 基本方針

- 単一巨大 compose ではなく「共通ベース + 系統別オーバーレイ」に分割する。
- `stack.sh` を統一入口にし、系統の違いは引数/設定ファイルで吸収する。
- 既存の `multi-container` / `all-in-one` は後方互換を維持する。

## 11.3 目標アーキテクチャ

- 共通:
  - `docker/compose.base.yml`（DB/Redis/MinIO/Newman/python-tools など共通基盤）
- 系統別:
  - `docker/compose.family.modern.yml`
  - `docker/compose.family.csharp.yml`
  - `docker/compose.family.java-legacy.yml`
  - 将来追加: `docker/compose.family.<family>.yml`
- 系統定義（拡張ポイント）:
  - `docker/families/registry.yml`（family 名、説明、compose 参照、推奨モード、必要環境変数）
- 実行入口:
  - `docker/stack.sh` に family 指定機能を追加（例: `--family modern|csharp|java-legacy|...`）

## 11.4 実装ステップ

1. 現行 `docker/compose.yml` を `compose.base.yml` + `compose.family.modern.yml` に分割（挙動差分ゼロで移行）
2. `stack.sh` に family 解決ロジックを追加し、compose ファイルの多重読み込みを標準化
3. 既存モード（`multi-container` など）を `modern` へマッピングして後方互換を維持
4. C# 系列の雛形 compose を追加（SDK 実行サービス + 任意で reverse proxy）
5. Java レガシー compose を追加（`apache` / `tomcat` 分離サービス）
6. `registry.yml` を導入し、4系統目以降の追加手順を固定化
7. Dev Container 側に family 切替手順を追加（共通 `devcontainer.json` + 系統別 override 方針）
8. ドキュメントに「新規 family の追加手順」を追記

## 11.5 運用インターフェース（案）

```bash
# 現行互換（暗黙 modern）
docker/stack.sh multi-container

# 明示的に family 指定
docker/stack.sh --family modern multi-container
docker/stack.sh --family csharp up
docker/stack.sh --family java-legacy up
```

## 11.6 受け入れ条件

- `modern` family で現行と同等の起動/テスト動線が維持される
- `csharp` / `java-legacy` family の雛形が `stack.sh` 経由で起動できる
- 新規 family 追加時に `compose.family.<name>.yml` + `registry.yml` 追記のみで拡張できる
- ドキュメントだけで第三者が family 追加と起動を再現できる

## 11.7 リスク

- compose 分割時に依存関係（`depends_on` / network / volume）を壊す可能性
- family ごとに命名規則が崩れると運用が属人化する
- Java レガシー系では `war` 配備前提と Spring Boot jar 前提が混在しやすい

## 11.8 採用決定（2026-04-09）

- `stack.sh` の family 指定方式は A + B 併用を採用する。
- 優先順位は `--family` > `STACK_FAMILY` > `modern`（デフォルト）とする。
- compose ファイル命名規則は当面 `compose.family.<name>.yml` を採用する。
- 将来、family 数や専用ファイル増加に応じて `families/<name>/compose.yml` への段階移行を検討する。
- family 運用は当面「排他的起動（同時起動しない）」を採用する。

## 13.4 Pact 導入計画

### 13.4.1 目的・用途

- FE ↔ BE、またはサービス間APIの契約（request/response）破壊を早期検知する。
- リリース前ではなく変更時点で仕様ズレを可視化する。

### 13.4.2 導入理由

- E2E/Newman だけでは「仕様互換性の差分」を網羅的に保証しにくいため。
- family横断で API 契約の共通品質ゲートを持てるため。

### 13.4.3 実装ステップ

1. 対象APIを限定して Pilot（認証API以外の低リスク領域）を実施（完了: `/test-cors`）

3. Pact Broker（または同等保存先）運用方針を決定（継続: 初期はリポジトリ内保存）
4. 認証前提APIはトークン取得を含む契約テスト設計に拡張（継続）
5. 段階的に対象APIを拡大（継続）

### 13.4.4 受け入れ条件

- Consumer/Provider 双方の CI で Pact 検証が通る
- 契約破壊時に MR 段階で検知できる
- 最低1ドメインの API 群で継続運用できる

## 13.5 OpenTelemetry Collector 導入計画

### 13.5.1 目的・用途

- ログ/メトリクス/トレースを統合し、障害調査時間を短縮する。

### 13.5.2 導入理由

- family が増えるほど、問題発生時の可観測性の差がボトルネックになるため。

5. 主要ユースケースのトレース確認手順をドキュメント化（完了）
