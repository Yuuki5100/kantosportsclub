# バックエンド 新規API配置・コーディング規約

> 文書ステータス: 廃止予定
> 位置づけ: ドメイン分割型の代替案を含む旧案。現行開発の判断根拠には使用しない
> 参照元: `開発ルール総則.md`

本規約は、現行のバックエンド構成（multi-module：gateway / appserver / batchserver / servercommon / syncconnector）を前提として、新規API・機能追加時の**配置場所およびパッケージ構成の統一**を目的とする。

現時点の正本は `バックエンドコーディング規約.md` とする。本書は将来的なドメイン分割型構成を検討した際の旧参考資料として残し、単独で配置判断に使用しない。

本書の記述は将来案であり、現行開発の必須ルールとしては扱わない。必要な内容は `バックエンドコーディング規約.md` 側へ移行済みのため、今後は `廃止・移行一覧.md` に従って参照停止を進める。

---

## 1. モジュール別の責務

### 1.1 gateway

* 認証・認可の入口、ルーティング、フィルタ処理を担う
* WebFlux ベース
* **業務ロジック・業務APIの実装は禁止**
* 許可される実装内容：

  * ルーティング設定
  * フィルタ（認証、ヘッダ付与、レート制限等）
  * 入口レベルの例外ハンドリング

### 1.2 appserver

* 業務APIの本体を担う
* Controller / Service / Repository / Security 等を実装
* **新規の業務APIは原則すべて本モジュールに実装する**

### 1.3 batchserver

* バッチ処理・スケジューラ・Spring Batch Job を担う
* 業務APIの実装は原則行わない（運用・管理用の最小限は例外）

### 1.4 servercommon

* 複数モジュールで共有される共通部品を格納する
* DTO / Entity / 例外 / ユーティリティ / テンプレート等
* **依存方向は servercommon → appserver にならないこと**

### 1.5 syncconnector

* 外部システム連携専用モジュール
* HTTP クライアント、署名、Interceptor 等を実装
* 外部API仕様に依存した DTO を保持する

## 2. 新規業務API追加時の基本ルール（appserver）

本章は将来的なドメイン分割型構成を採る場合の参考案である。現行開発では、本章の内容より `バックエンドコーディング規約.md` を優先する。

### 2.1 基本パッケージ構成

* Controller：`com.example.appserver.controller`
* Service：`com.example.appserver.service`
* Repository：将来案として `com.example.appserver.repository`
* Config：`com.example.appserver.config`
* Security：`com.example.appserver.security`

### 2.2 機能（ドメイン）単位でのサブパッケージ化

新規APIは**機能名（ドメイン名）単位でまとめる**こと。

例：reservation（予約）機能の場合

* Controller：`controller.reservation`
* Service：`service.reservation`
* Repository：`repository.reservation`
* Request DTO：`request.reservation`
* Response DTO：`response.reservation`

※ admin / auth 等の横断カテゴリ配下に無秩序に追加しないこと

---

## 3. Request / Response DTO の配置ルール

### 3.1 原則

* 当該APIのみで使用する DTO：appserver 配下に配置
* 複数モジュールで共有する DTO：servercommon に配置

### 3.2 servercommon に配置してよい条件

以下のいずれかを満たす場合のみ許可する。

* appserver と batchserver（または syncconnector）で共通利用される
* 将来的な共通利用が確実で、依存方向が破綻しない

※ 判断に迷う場合は **appserver に配置すること**

---

## 4. 外部連携機能追加時の配置ルール（syncconnector）

### 4.1 syncconnector 側

* HTTP クライアント：`syncconnector.http`
* Interceptor：`syncconnector.interceptor`
* 署名ロジック：`syncconnector.signature`
* 外部API用 DTO：`syncconnector.model.<feature>`

### 4.2 appserver 側（呼び出し元）

* 外部連携の呼び出しは Service 層から行う
* appserver の DTO と外部DTOは**必ず変換する**
* 外部DTOを Controller から直接返却しない

---

## 5. バッチ追加時の配置ルール（batchserver）

* Job 定義：`batchserver.jobs.<jobName>`
* Tasklet：`batchserver.tasklet.<jobName>`
* Listener：`batchserver.listener`
* Scheduler：`batchserver.scheduler`
* バッチ専用モデル：`batchserver.model.<jobName>`

共通化が必要な DTO / モデルは servercommon に移動すること。

---

## 6. 命名規約

* Controller：`XxxController`
* Service：`XxxService` / `XxxServiceImpl`
* Request DTO：`XxxRequest`
* Response DTO：`XxxResponse`
* 外部連携 Client：`XxxClient`
* 外部API DTO：`XxxApiRequest` / `XxxApiResponse`

---

## 7. 新規API追加時の最小構成例

例：`/api/reservations` を追加する場合

* appserver

  * `controller/reservation/ReservationController`
  * `service/reservation/ReservationService`
  * `request/reservation/CreateReservationRequest`
  * `response/reservation/ReservationResponse`
  * `repository/reservation/ReservationRepository`（将来案）

* servercommon（必要な場合）

  * `dto/ReservationDto`

* syncconnector（外部連携がある場合）

  * `http/PaymentsClient`
  * `model/payments/...`

---

本規約に従い、新規API・機能追加時は**配置場所・責務・依存方向の一貫性**を必ず維持すること。

ただし、現行開発での正式基準は `バックエンドコーディング規約.md` とする。
