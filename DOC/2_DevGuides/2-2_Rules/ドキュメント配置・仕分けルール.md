# ドキュメント配置・仕分けルール

> 文書ステータス: 正本
> 位置づけ: `DOC` 配下の文書をどこへ配置するかを定義する正本ルール
> 参照元: `開発ルール総則.md`

## 1. 目的

本書は、`DOC` 配下の文書構成と配置基準を明確化し、既存文書の移行と新規文書の追加を同じルールで運用できるようにするための正本である。

特に以下を目的とする。

- 人が見ても迷わず文書の置き場を判断できること
- agent が既存文書を機械的に仕分けできること
- 設計書、手順書、規約、テスト資料、調査メモの混在を防ぐこと

---

## 2. 基本原則

- 文書はタイトルではなく「主目的」で分類する
- 1文書1責務を原則とし、複数カテゴリにまたがる場合は分割を優先する
- 現在の正本として参照する文書と、調査・比較・生成物は分離する
- `DOC` 直下には `README.md` 以外を新規配置しない
- `1_DesignDocument/1-1_BaseDocs` 直下には索引・分類基準以外を原則配置しない
- 仮名称 (`rename.md`, `tmp.md`, `draft.md`) のまま正規配置しない

---

## 3. 判定の優先順位

配置先は、以下の順で判断する。

1. 文書の主目的
2. 成果物の性質（正本 / 調査 / 差分 / 自動生成 / テスト設計 / テスト実行）
3. 主たる利用者（設計者 / 開発者 / QA / 運用者）
4. 対象領域（BE / FE / 共通 / Git / CI/CD）
5. ファイル名や拡張子

補足:

- タイトルに `設計書` とあっても、中身が手順書であれば `2_DevGuides` に置く
- タイトルに `ガイド` とあっても、中身がシステム仕様であれば `1_DesignDocument` に置く
- 文書名だけで判断できない場合は、冒頭 30 行と見出し構成を優先して判定する

## 5. `1_DesignDocument` の仕分けルール

### 5.1 `1-1_BaseDocs` / `1-2_GeneratedDocs` / `1-3_Diffs` / `1-4_TestDesignDocs`

| 配置先 | 判定基準 |
|--------|----------|
| `1-1_BaseDocs` | 現在のシステム仕様・設計として継続参照する文書 |
| `1-2_GeneratedDocs` | 手作業で正本管理せず、ツールやスクリプトで再生成できる文書 |
| `1-3_Diffs` | 2つ以上の対象を比較し、差分や移行判断を残すことが主目的の文書 |
| `1-4_TestDesignDocs` | 設計仕様から導出したテスト設計、観点設計、試験方針 |

以下は `1-3_Diffs` を優先する。

- 旧PJと現PJの比較
- 派生PJからの移行差分
- As-Is / To-Be 比較
- 実装差異棚卸し

以下は `1-4_TestDesignDocs` を優先する。

- 機能別テスト設計書
- API結合観点設計
- 試験方針と観点定義

---

### 5.2 `1-1_BaseDocs/BE` の配置先

`BE` 配下は、`DOC/1_DesignDocument/1-1_BaseDocs/BE/00_目次/info.md` の分類基準を正本として扱い、以下の通り配置する。

| 配置先 | 置くもの |
|--------|----------|
| `BE/00_目次` | 目次、分類基準、索引 |
| `BE/01_全体構成` | サーバー構成、責務分担、基盤構成、処理方式、全体像 |
| `BE/02_設定値` | `application.yml`、環境差分、設定一覧、設定値定義 |
| `BE/03_共通部品` | 複数機能で使う基盤部品、共通サービス、ユーティリティ |
| `BE/04_共通規約` | API命名、認可ルール、共通設計ルール |
| `BE/05_個別機能` | 業務機能単位、API単位の設計書 |
| `BE/06_外部連携` | 外部システム、外部サービス、外部I/F連携仕様 |
| `BE/07_例外処理` | 例外方針、エラー応答、失敗時挙動、エラーハンドリング |
| `BE/08_状態_コード定義` | Enum、状態遷移、コード定義、テーブル定義 |
| `BE/09_調査_課題` | 調査メモ、差異分析、課題整理、改善案、実装計画 |
| `BE/reuse` | 再利用判断基準、再利用台帳、依存関係、実装パターン |

BE の強制ルール:

- 設定値を含む設計書は、設定一覧だけを `BE/02_設定値` に分離する
- Enum や状態コードが主題の資料は `BE/08_状態_コード定義` に置く
- 外部連携仕様は、個別機能設計書から分離して `BE/06_外部連携` に置く
- 調査・課題・改善提案は、正本設計書に混在させず `BE/09_調査_課題` に置く

---

### 5.3 `1-1_BaseDocs/FE` の配置先

`FE` 配下は、`DOC/1_DesignDocument/1-1_BaseDocs/FE/README.md` の分類基準を正本として扱い、以下の通り配置する。

| 配置先 | 置くもの |
|--------|----------|
| `FE/00_overview` | 総覧、入口資料、全体目次、索引 |
| `FE/01_architecture` | ディレクトリ構成、状態管理、環境変数、i18n、設計思想 |
| `FE/02_application-foundation` | 認証、API通信、エラー処理、ログ、通知などの横断基盤 |
| `FE/03_realtime` | WebSocket 等のリアルタイム通信設計 |
| `FE/04_features` | 業務機能、画面機能、機能別UI仕様 |
| `FE/modules/base` | 最小単位の汎用UI部品 |
| `FE/modules/composite` | 複数部品を組み合わせた共通UI |
| `FE/modules/functional` | 業務ロジックやAPI依存を持つ機能コンポーネント |
| `FE/modules/providers` | 全体注入系の provider |
| `FE/modules/examples` | サンプル、使用例、デモ用途 |
| `FE/reuse` | 再利用判断基準、再利用台帳、依存関係、判断フロー |

FE の強制ルール:

- UI部品ドキュメントは、画面機能資料と混在させず `modules` 配下へ置く
- サンプルや利用例は本番設計と分離し `modules/examples` に置く
- WebSocket などのリアルタイム通信は機能資料から分離し `03_realtime` に置く
- 再利用判断用の横断資料は `reuse` に集約する

---

### 5.4 BE / FE のどちらに置くか迷う場合

以下の順で判断する。

1. 主たる実装責務がサーバー側なら `BE`
2. 主たる実装責務が画面側なら `FE`
3. 同じ比重で両方を扱う場合は、1文書に閉じ込めず BE/FE に分割する
4. 分割前提で一時的に保留する場合は、正規配置せず `要分割` として扱う

補足:

- API仕様そのものは `BE` を正本とする
- API利用方法やフロント呼び出し規約は `FE`
- 画面仕様の中にAPI一覧がある場合、API定義は `BE` 正本へ寄せ、FE には参照リンクのみ残す

## 6. `2_DevGuides` の仕分けルール

| 配置先 | 置くもの | 代表例 |
|--------|----------|--------|
| `2-1_HowToDevelop` | 環境構築、起動方法、ローカル開発手順 | WSL設定、devcontainer、起動手順 |
| `2-2_Rules` | 規約、標準、命名、責任体制、レビュー/例外運用 | コーディング規約、命名規則、文書ルール |
| `2-3_DeveloperGuide` | 実装の具体手順、開発フローの実務ガイド | API追加手順、通信実装手順 |
| `2-4_GitDocs` | Git / SVN / Subtree / 履歴移行 | SVN移行、authors生成 |
| `2-5_CiCd` | CI/CD、ビルド、デプロイ、検査ジョブ | deploy、カバレッジ、FindSecBug |

判定ルール:

- 「守るべき基準」を定義する文書は `2-2_Rules`
- 「作業の進め方」を説明する文書は `2-3_DeveloperGuide`
- 「環境を作る」「起動する」文書は `2-1_HowToDevelop`
- 「パイプラインやデプロイ」を扱う文書は `2-5_CiCd`

## 7. `3_TestQa` の仕分けルール

`3_TestQa` は、テスト実行や QA 運用に使う文書の置き場とする。

| 配置先 | 置くもの |
|--------|----------|
| `3-1_Testing` | テストケース、観点表、実施手順、証跡テンプレート、チェックリスト、テスト支援ツール説明 |

`1-4_TestDesignDocs` との違い:

- 設計書から導いた試験設計は `1-4_TestDesignDocs`
- 実際の試験実施、証跡、チェック運用、ユーザー確認資料は `3_TestQa`

## 8. agent 用の仕分け手順

agent は、以下の手順で文書の配置先を決定する。

1. 文書タイトル、冒頭、見出しを読み、主目的を 1 つだけ決める
2. 主目的が「設計」「開発運用」「テスト実施」のどれかを判定する
3. `1_DesignDocument` / `2_DevGuides` / `3_TestQa` のいずれかを選ぶ
4. 設計文書であれば、正本 / 自動生成 / 差分 / テスト設計を判定する
5. 対象が BE / FE / 共通開発 / Git / CI/CD のどれかを判定する
6. 最後に配下カテゴリへ配置する

自動配置してよい条件:

- 主目的が明確
- 既存の分類表のどれかに一致する
- 1文書1責務が保たれている

自動配置してはいけない条件:

- 1文書の中に設計書、手順書、調査メモが混在している
- BE と FE が同等比重で混在している
- 仮名称で正式な文書名が確定していない
- 文書の正本か調査メモかが判別できない

自動配置不可の場合の扱い:

- 正規配置せず、候補パスと判断理由を記録する
- 必要なら文書を分割し、分割後にそれぞれ正規配置する

## 10. 既存文書移行時の追加ルール

- 旧PJ由来の比較資料は、正本へ直接混ぜず `1-3_Diffs` へ寄せる
- 現行化済みの設計内容だけを残す場合は、比較記述を削除して `1-1_BaseDocs` に置く
- `rename.md` のような暫定ファイル名は、内容に応じた正式名称へ変更してから配置する
- テンプレート文書は、使用先カテゴリに置く
- prompt や実装指示書は、再利用可能な運用ルールなら `2_DevGuides`、特定の調査テーマ専用ならその調査配下に置く

---

## 11. 更新ルール

- 新しい文書種別や配置先を追加する場合は、本書を先に更新する
- `DOC/README.md` の概要構成と、本書の配置基準は同時に整合させる
- `BE` / `FE` 個別の分類基準を変更する場合は、本書と各入口文書 (`BE/00_目次/info.md`, `FE/README.md`) を同時に更新する
- 文書オーナー変更時は `文書オーナー一覧.md` を同一変更で更新する
