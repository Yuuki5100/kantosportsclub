# フロントエンド開発コーディング規約

> 文書ステータス: 正本
> 位置づけ: フロントエンド開発の主規約
> 参照元: `開発ルール総則.md`

## 概要

本ドキュメントは、my-next-appをベースとしたフロントエンド開発の委託先向けコーディング規約です。
品質と保守性を確保するため、以下の規約に従って開発を行ってください。

本書では次の語をこの意味で用いる。

- `必須`: 例外申請がない限り守る
- `推奨`: 標準の実装方法として採用する
- `任意`: チーム判断で採用できる
- `禁止`: 新規実装では採用しない

### ファイル命名規則

- `src` 直下や `components` 配下のグループディレクトリは lower-case を基本とする
- 具体コンポーネントのディレクトリやコンポーネントファイルは PascalCase を基本とする
- **コンポーネントファイル**: PascalCase（例: `UserProfile.tsx`）
- **フックファイル**: camelCase かつ export 名と一致（例: `useUserData.ts`, `useCurrentLanguage.ts`, `useValidationLang.ts`）
- **ユーティリティファイル**: camelCase（例: `formatDate.ts`）
- **langファイル**: `camelCase + .lang.ts`（例: `login.lang.ts`, `fileValidation.lang.ts`）
- **型定義ファイル**: camelCase（例: `userTypes.ts`）
- **定数ファイル**: camelCase（例: `apiEndpoints.ts`）
- **テストファイル**: `*.test.tsx`
- **Storybookファイル**: `*.stories.tsx`

### i18n / Hook 命名の標準

- 言語状態の参照・更新は `useCurrentLanguage` を使用する
- 辞書オブジェクトから表示文言を選ぶ Hook は `useLanguage` を使用する
- validation 系の言語選択 Hook は `useValidationLang` を使用する
- `lang` ファイルと Hook ファイルに例外命名を作らない

## TypeScript 規約

### 型定義

#### 1. 型命名規則

**基本方針**: `type`キーワードをデフォルトとし、継承・Declaration Mergingが必要な場合は`interface`を検討する

```tsx
// ✅ 良い例 - typeをデフォルト使用
type UserProfile = {
  id: string;
  name: string;
  email: string;
};

type UserRole = 'admin' | 'user' | 'guest';

// Props型には接尾辞 'Props' を付ける（typeが自然）
type UserCardProps = {
  user: UserProfile;
  onEdit: (id: string) => void;
};

// Union型やIntersection型（typeが適している）
type Status = 'loading' | 'success' | 'error';
type Theme = 'light' | 'dark';
type BaseProps = { id: string };
type ExtendedProps = BaseProps & { name: string }; // Intersection

// ❌ 悪い例
interface IUser {  // I接頭辞は使わない
  id: string;
}

interface userProfile {  // PascalCaseを使う
  id: string;
}
```

**interfaceを使用するケース**:

1. **継承が必要な場合（ジェネリクス含む）**
2. **ライブラリの型拡張が必要な場合（Declaration Merging）**
3. **ライブラリとの互換性が必要な場合**

**重要**: Declaration Mergingは**ライブラリの型拡張でのみ許可**し、アプリケーション内での使用は原則禁止

```tsx
// ✅ interfaceが適している場合

// MUIテーマの型拡張
declare module '@mui/material/styles' {
  interface Theme {
    customColors: {
      primary: string;
      secondary: string;
    };
  }
  
  interface ThemeOptions {
    customColors?: {
      primary?: string;
      secondary?: string;
    };
  }
}

// Next.jsの型拡張
declare module 'next' {
  interface NextApiRequest {
    user?: {
      id: string;
      role: string;
    };
  }
}

// React関連の型拡張
declare module 'react' {
  interface CSSProperties {
    '--custom-property'?: string;
  }
}

// ❌ 悪い例 - アプリケーション内でのDeclaration Mergingは禁止
// 意図しないマージの原因になるため避ける
interface UserProfile {
  id: string;
  name: string;
}

// 別ファイルで同名interfaceを定義
interface UserProfile {
  email: string;  // 予期しない型の変更が起こる可能性がある
}

// ✅ 良い例 - アプリケーション内ではtypeを使用し、型の拡張は明示的に行う
type BaseUserProfile = {
  id: string;
  name: string;
};

type ExtendedUserProfile = BaseUserProfile & {
  email: string;
};

// 3. ライブラリとの互換性（Material-UIなど）
interface CustomTheme extends Theme {
  customPalette: {
    accent: string;
  };
}
```

#### 2. 型のエクスポート

```tsx
// ✅ 良い例 - 型は名前付きエクスポート（typeを使用）
export type UserProfile = {
  id: string;
  name: string;
};

export type UserRole = 'admin' | 'user';

// ❌ 悪い例 - 型のデフォルトエクスポートは避ける
type UserProfile = {
  id: string;
}
export default UserProfile;

// ❌ 悪い例 - 特別な理由がない限りinterfaceは使用しない
export interface UserProfile {
  id: string;
}
```

#### 3. 厳密な型定義

```tsx
// ✅ 良い例 - typeを使用
type ApiResponse<T> = {
  data: T;
  status: 'success' | 'error';
  message?: string;
};

// ❌ 悪い例 - any の使用は避ける
type ApiResponse = {
  data: any;  // 具体的な型を定義する
  status: string;  // Union型で制限する
};
```

### クラス設計規約

**基本方針**: 関数型プログラミングを優先し、`class`は特別な理由がない限り使用しない
**重要**: `class`を検討する場合は、まず**Factory関数**での実装を検討すること

#### Factory関数の優先採用

オブジェクト指向的な設計が必要な場合、まずはFactoryパターンでの実装を検討する。

```tsx
// ✅ 推奨 - Factory関数パターン
type UserService = {
  getUsers: () => Promise<User[]>;
  createUser: (user: CreateUserRequest) => Promise<User>;
  updateUser: (id: string, user: UpdateUserRequest) => Promise<User>;
  // プライベートな状態も表現可能
  readonly config: ServiceConfig;
};

export const createUserService = (apiClient: ApiClient): UserService => ({
  getUsers: () => apiClient.get('/users'),
  createUser: (user) => apiClient.post('/users', user),
  updateUser: (id, user) => apiClient.put(`/users/${id}`, user),
});

// React カスタムフックでの使用
export const useUserService = () => {
  const apiClient = useApiClient();
  const config = useServiceConfig();
  return useMemo(() => createUserService(apiClient, config), [apiClient, config]);
};
```

#### Factory関数の利点

1. **関数型プログラミングとの親和性**
2. **テストのしやすさ**: 依存関係の注入が容易
3. **イミュータブルな設計**: 状態変更を制御しやすい
4. **TypeScriptの型推論**: classよりも型安全
5. **Tree Shaking**: 未使用コードの除去が効率的

#### classを使用する例外ケース

Factory関数では実現困難な場合のみclassを使用する：

```tsx
// ✅ 例外的にclassが適している場合
// - React Class Componentの拡張（既存コードの場合）
// - 外部ライブラリとの互換性要求
// - 複雑な継承階層が必要な場合

// ❌ 悪い例 - Factory関数で実現可能
class UserService {
  constructor(private apiClient: ApiClient) {}
  
  async getUsers(): Promise<User[]> {
    return this.apiClient.get('/users');
  }
}

// ✅ 良い例
// - 外部ライブラリやErrorの継承が必要な場合
class CustomError extends Error {
  constructor(
    message: string,
    public code: string,
    public statusCode: number
  ) {
    super(message);
    this.name = 'CustomError';
  }
};
```

## コンポーネント設計規約

### 1. Props設計規約

#### Props型定義の必須規則

**基本方針**:

- 関数コンポーネントのProps型は`type`キーワードをデフォルトとする
- 継承・Declaration Merging・ライブラリ互換性などの都合がある場合、`interface`を検討する
- 特別な理由がない限り、関数コンポーネントでのインライン型定義は禁止とする
- Props型名は`コンポーネント名 + Props`とする

```tsx
// ✅ 良い例 - 基本的なProps型定義（typeを使用）
type UserCardProps = {
  user: UserProfile;

  isSelected?: boolean;
  hasPermission?: boolean;
  canEdit?: boolean;
  shouldShowActions?: boolean;
  
  onEdit?: (userId: string) => void;
  onDelete?: (userId: string) => void;

  className?: string;
  children?: React.ReactNode;
};

// Union/Intersection型の使用
type ButtonVariant = 'primary' | 'secondary' | 'danger';

type BaseButtonProps = {
  children: React.ReactNode;
  onClick: () => void;
};

type ButtonProps = BaseButtonProps & {
  variant: ButtonVariant;
  size?: 'small' | 'medium' | 'large';
};

type BaseListProps<T> = {
  items: T[];
  loading?: boolean;
  onItemClick?: (item: T) => void;
}

type UserListProps = BaseListProps<UserProfile> & {
  showAvatar?: boolean;
  sortBy?: keyof UserProfile;
}

// 関数コンポーネントでの使用
export const UserCard: React.FC<UserCardProps> = ({ user, isSelected, onEdit }) => {
  // ...コンポーネント実装
};

export const UserList: React.FC<UserListProps> = ({ items, showAvatar, onItemClick }) => {
  // ...コンポーネント実装
};

// ❌ 悪い例 - インライン型定義は禁止
export const UserCard: React.FC<{
  user: UserProfile;
  isSelected?: boolean;
}> = ({ user, isSelected }) => {
  // ...
};
```

#### 必須・任意の明確化

```tsx
// ✅ 良い例 - typeを使用
type ComponentProps = {
```

## 単体テスト規約

### コンポーネントテスト

- 適切なレンダリングが行われているかテストすること
- 複雑なロジックがある場合、関数として切り出してテストすること

#### 複雑なロジックの関数切り出しサンプル

**悪い例**: コンポーネント内に複雑なロジックを記述

```tsx
// ❌ 悪い例 - テストが困難
const UserManagementComponent = ({ users }: { users: User[] }) => {
  const [filteredUsers, setFilteredUsers] = useState<User[]>([]);

  useEffect(() => {
    // 複雑なフィルタリングロジックがコンポーネント内にある
    const filtered = users.filter(user => {
      // 複雑な条件判定
      const isActive = user.status === 'active';
      const hasValidEmail = user.email && /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(user.email);
      const isAdult = user.age >= 18;
      const hasRecentActivity = user.lastLoginDate && 
        new Date(user.lastLoginDate) > new Date(Date.now() - 30 * 24 * 60 * 60 * 1000);
      
      return isActive && hasValidEmail && isAdult && hasRecentActivity;
    }).sort((a, b) => {
      // 複雑なソートロジック
      const scoreA = calculateUserScore(a);
      const scoreB = calculateUserScore(b);
      return scoreB - scoreA;
    });

    setFilteredUsers(filtered);
  }, [users]);

  // 複雑な計算ロジック
  const calculateUserScore = (user: User) => {
    let score = 0;
    if (user.premiumMember) score += 100;
    if (user.referralCount > 5) score += user.referralCount * 10;
    if (user.purchaseHistory.length > 0) {
      score += user.purchaseHistory.reduce((sum, purchase) => sum + purchase.amount, 0) / 100;
    }
    return score;
  };

  return <UserList users={filteredUsers} />;
};
```

**良い例**: 複雑なロジックを純粋関数として切り出し

```tsx
// ✅ 良い例 - ロジックを分離してテスト可能に

// utils/userFilters.ts - 純粋関数として切り出し
export const isValidUser = (user: User): boolean => {
  const isActive = user.status === 'active';
  const hasValidEmail = user.email && /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(user.email);
  const isAdult = user.age >= 18;
  const hasRecentActivity = user.lastLoginDate && 
    new Date(user.lastLoginDate) > new Date(Date.now() - 30 * 24 * 60 * 60 * 1000);
  
  return isActive && hasValidEmail && isAdult && hasRecentActivity;
};

export const calculateUserScore = (user: User): number => {
  let score = 0;
  if (user.premiumMember) score += 100;
  if (user.referralCount > 5) score += user.referralCount * 10;
  if (user.purchaseHistory.length > 0) {
    score += user.purchaseHistory.reduce((sum, purchase) => sum + purchase.amount, 0) / 100;
  }
  return score;
};

export const filterAndSortUsers = (users: User[]): User[] => {
  return users
    .filter(isValidUser)
    .sort((a, b) => calculateUserScore(b) - calculateUserScore(a));
};

// components/UserManagementComponent.tsx - シンプルなコンポーネント
const UserManagementComponent = ({ users }: { users: User[] }) => {
  const filteredUsers = useMemo(() => filterAndSortUsers(users), [users]);
  return <UserList users={filteredUsers} />;
};
```

**テストコード例**:

```tsx
// __tests__/userFilters.test.ts - 純粋関数の単体テスト
import { isValidUser, calculateUserScore, filterAndSortUsers } from '../utils/userFilters';
import { User } from '../types';

describe('userFilters', () => {
  const validUser: User = {
    id: '1',
    status: 'active',
    email: 'test@example.com',
    age: 25,
    lastLoginDate: new Date().toISOString(),
    premiumMember: false,
    referralCount: 3,
    purchaseHistory: [],
  };

  describe('isValidUser', () => {
    test('should return true for valid user', () => {
      expect(isValidUser(validUser)).toBe(true);
    });

    test('should return false for inactive user', () => {
      const inactiveUser = { ...validUser, status: 'inactive' };
      expect(isValidUser(inactiveUser)).toBe(false);
    });

    test('should return false for user with invalid email', () => {
      const invalidEmailUser = { ...validUser, email: 'invalid-email' };
      expect(isValidUser(invalidEmailUser)).toBe(false);
    });

    test('should return false for underage user', () => {
      const underageUser = { ...validUser, age: 17 };
      expect(isValidUser(underageUser)).toBe(false);
    });
  });

  describe('calculateUserScore', () => {
    test('should calculate score for premium member', () => {
      const premiumUser = { ...validUser, premiumMember: true };
      expect(calculateUserScore(premiumUser)).toBe(100);
    });

    test('should add referral bonus for high referral count', () => {
      const referralUser = { ...validUser, referralCount: 10 };
      expect(calculateUserScore(referralUser)).toBe(100); // 10 * 10
    });

    test('should add purchase history bonus', () => {
      const purchaseUser = {
        ...validUser,
        purchaseHistory: [{ amount: 1000 }, { amount: 500 }]
      };
      expect(calculateUserScore(purchaseUser)).toBe(15); // (1000 + 500) / 100
    });
  });

  describe('filterAndSortUsers', () => {
    test('should filter and sort users correctly', () => {
      const users = [
        { ...validUser, id: '1', premiumMember: true }, // score: 100
        { ...validUser, id: '2', referralCount: 10 }, // score: 100
        { ...validUser, id: '3', status: 'inactive' }, // filtered out
        { ...validUser, id: '4', referralCount: 3 }, // score: 0
      ];

      const result = filterAndSortUsers(users);
      
      expect(result).toHaveLength(3);
      expect(result[0].id).toBe('1'); // highest score first
      expect(result[1].id).toBe('2');
      expect(result[2].id).toBe('4');
    });
  });
});
```

**関数切り出しのメリット**:
1. **単体テストが容易** - 純粋関数なので入力と出力の関係が明確
2. **再利用性が高い** - 他のコンポーネントからも利用可能
3. **保守性の向上** - ロジックの変更時の影響範囲が限定される
4. **可読性の向上** - コンポーネントがシンプルになる
5. **デバッグが容易** - 関数単位での動作確認が可能

#### SSR（Server-Side Rendering）の使用方針

**基本方針**: toBシステムのため、**全ページでCSR（ssr: false）を推奨**

```tsx
// ✅ toBシステムでの推奨実装（全ページCSR）
const BusinessComponent = dynamic(
  () => import('./BusinessComponent'),
  {
    ssr: false, // toBシステムでは全ページCSR推奨
    loading: () => <div>読み込み中...</div>
  }
);
```

**CSR（ssr: false）を推奨する理由**:

- **開発効率**: ブラウザ専用APIを制約なしで使用可能
- **保守性**: テスト容易性・デプロイ容易性を維持する
- **SEO不要**: CRJはtoB向けのため、検索エンジン最適化は不要
- **キャッシュ効果**: 一度ロードしたJSは再利用され、後続ページが高速表示

```tsx
// ✅ 良い例 - 適切な検証
import { z } from 'zod';
```

### 2. XSS対策とHTML使用制限

**基本方針**: 生のHTMLの使用は原則禁止。Reactの標準的なJSX記法を使用する。

```tsx
// ✅ 良い例 - JSXによる安全なレンダリング
export const SafeContent = ({ title, description }: { title: string; description: string }) => {
  return (
    <div>
      <h2>{title}</h2>
      <p>{description}</p>
    </div>
  );
};

// ✅ 良い例 - MUIコンポーネントの使用
export const FormattedText = ({ content }: { content: string }) => {
  return (
    <Typography variant="body1">
      {content}
    </Typography>
  );
};

// ❌ 悪い例 - dangerouslySetInnerHTML の使用は禁止
export const UnsafeHtml = ({ html }: { html: string }) => {
  return <div dangerouslySetInnerHTML={{ __html: html }} />; // XSSリスク・使用禁止
};

// ❌ 悪い例 - innerHTML の直接操作も禁止
export const UnsafeDOMManipulation = () => {
  useEffect(() => {
    document.getElementById('content')!.innerHTML = userInput; // XSSリスク・使用禁止
  }, []);
  
  return <div id="content" />;
};
```

#### HTMLの直接使用禁止

**本プロジェクトでは、一部画面を除きリッチテキスト入力や外部HTML読み込みの要件がないため、XSSリスクを完全に排除します。**

**具体的な禁止事項**:

- `dangerouslySetInnerHTML` の使用
- `innerHTML` の直接操作
- 外部HTMLの読み込み・表示
- ユーザー入力のHTML形式での処理

**理由**: 本プロジェクトでは、一切画面を除きリッチテキスト入力や外部HTML読み込みの機能がないため、XSSリスクを完全に排除します。

## メッセージ管理規約

### 1. MessageCatalog の利用（必須）

- 画面に表示するエラー・警告・バリデーション・完了メッセージは `src/message/index.ts` で一元管理する
- 表示時は `getMessage` と `MessageCodes` を使用する
- API由来のメッセージを表示する場合でも、フォールバックは MessageCatalog を使用する
- 生文字列のエラーメッセージを `showError` / `showSnackbar` / `handleApiError` / `new Error` に直接渡さない（ESLintで検知）

### 2. 実装例

```tsx
import { getMessage, MessageCodes } from '@/message';
import { useSnackbar } from '@/hooks/useSnackbar';

const { showSnackbar } = useSnackbar();

// ✅ 良い例: MessageCatalog を利用
showSnackbar(getMessage(MessageCodes.UPLOAD_SUCCESS), 'SUCCESS');
showSnackbar(getMessage(MessageCodes.FETCH_FAILED), 'ALERT');
throw new Error(getMessage(MessageCodes.UNEXPECTED_ERROR));

// ✅ 良い例: プレースホルダ置換
const msg = getMessage(MessageCodes.CSV_HEADER_MISSING).replace('{missing}', missing.join(', '));

// ❌ 悪い例: 生文字列の直接指定
showSnackbar('アップロードに失敗しました', 'ALERT');
throw new Error('必須項目です');
```
