# 🧭 命名規則・スタイルガイド（BE/FE共通）

> 文書ステータス: 補足
> 位置づけ: BE/FE 共通の命名補助指針。個別規約と衝突時は個別規約を優先する
> 参照元: `開発ルール総則.md`

本ガイドは、**バックエンド（Spring Boot）およびフロントエンド（Next.js）** における命名規則、コメントスタイル、i18n対応方針を統一し、保守性・可読性・自動検証の整合性を高めるための共通ルールを定義します。

本書では次の語をこの意味で用いる。

- `必須`: 例外申請がない限り守る
- `推奨`: 標準の実装方法として採用する
- `任意`: チーム判断で採用できる
- `禁止`: 新規実装では採用しない

---

## 1. 命名規則（BE/FE共通）

### 1.1 基本方針

| 対象 | 命名規則 | 例 |
|------|----------|----|
| クラス名 | `PascalCase` | `UserService`, `LoginController` |
| メソッド・関数 | `camelCase`（動詞始まり） | `getUserData`, `submitForm` |
| 変数 | `camelCase` | `formState`, `errorMessage` |
| 定数 | `UPPER_SNAKE_CASE`（グローバル） / `camelCase`（ローカル） | `MAX_RETRY_COUNT`, `defaultTimeout` |
| Enum | クラス名: `PascalCase` / 値: `UPPER_SNAKE_CASE` | `UserStatus.ACTIVE` |
| Reactフック名 | `use` + `PascalCase` | `useAuth`, `useSnackbar` |

---

### 1.2 フロントエンド命名（Next.js）

フロントエンド命名は、現行実装を基準として以下を採用する。

| 対象 | 命名規則 | 例 |
|------|----------|----|
| `src` 直下の機能分類ディレクトリ | `lowercase` | `components`, `hooks`, `pages`, `message`, `utils` |
| `components` 配下のグループディレクトリ | `lowercase` を基本 | `base`, `composite`, `functional`, `providers` |
| 具体コンポーネントのディレクトリ名 | `PascalCase` を基本 | `Header`, `Footer`, `FileUpload` |
| コンポーネントファイル名 | `PascalCase` を基本 | `Header.tsx`, `UserCard.tsx` |
| Hookファイル | `camelCase` かつ export 名と一致 | `useCurrentLanguage.ts`, `useValidationLang.ts`, `useSnackbar.ts` |
| langファイル | `camelCase + .lang.ts` | `header.lang.ts`, `login.lang.ts`, `fileValidation.lang.ts` |
| indexファイル | 固定名 `index.ts`, `index.tsx` | |
| テストファイル | `*.test.ts(x)` | `Header.test.tsx`, `authSlice.test.ts` |
| 型定義ファイル | `camelCase` または `*.types.ts` | `userTypes.ts`, `user.types.ts` |

---

### 1.3 バックエンド命名（Spring Boot）

| 対象 | 命名規則 | 例 |
|------|----------|----|
| パッケージ名 | 小文字・ドット区切り | `com.example.service.user` |
| サービス/実装 | `UserService`, `UserServiceImpl` | インターフェースと実装で分離 |
| DTO/Entity/VO | `UserRequest`, `UserResponse`, `UserEntity` | サフィックスを明示 |

## 2. コメントスタイル

### 2.1 JSDoc / JavaDocスタイルの推奨

```ts
/**
 * ユーザー情報を取得するAPI
 * @param userId - 対象ユーザーのID
 * @returns ユーザー情報
 */
function fetchUser(userId: string): Promise<User> { ... }
```

### 2.2 補足・注意コメント

- `// TODO:`: 実装予定タスク
- `// FIXME:`: 一時的な対処やバグ
- `// NOTE:`: 補足情報や設計意図

## 3. i18n対応方針（Next.js）

### 3.1 langファイル構成

- 各機能ごとに `{module}.lang.ts` を作成する
- `lang` ファイルは `.lang.ts` サフィックスを必須とする
- 言語キー `ja` / `en` を持つオブジェクト形式

```ts
export const headerLang = {
  ja: { title: 'ヘッダー' },
  en: { title: 'Header' }
};
```

### 3.2 コンポーネントでの使用

```ts
const { language } = useCurrentLanguage();
<Text>{headerLang[language].title}</Text>
```

- 言語状態の参照・更新は `useCurrentLanguage` を使用する
- 辞書オブジェクトから表示文言を選ぶ Hook は `useLanguage(langObj)` を使用する
- validation 系の言語選択 Hook は `useValidationLang()` を使用する

### 3.3 メッセージキーの命名方針

- 機能 + 意味単位（例：`button`, `label`, `error`）
- ネストは2階層まで推奨（可読性を考慮）

## 4. 静的検証（ESLint + VSCode）

### 設定例（FE）

以下は命名規則を静的検証へ落とし込む際のイメージであり、実際の設定値はプロジェクトの ESLint 構成に合わせる。

```json
{
  "rules": {
    "filename-case": [
      "error",
      {
        "componentFiles": "PascalCase",
        "langFiles": "camelCase + .lang.ts",
        "groupDirectories": "lowercase"
      }
    ]
  }
}
```

### `.vscode/settings.json`

```json
{
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": true
  },
  "eslint.validate": ["typescript", "typescriptreact"],
  "editor.formatOnSave": false
}
```

## 5. Git / OS間対策（Windows ⇔ Linux）

```bash
git config --global core.ignorecase false
```

- 大文字小文字の区別を有効化
- Windows上で大文字小文字のみを変更する場合は、`別名 → 目的名` の2段階リネームが必要なことがある

---

## 6. 運用ルール

- コードレビュー時に命名・i18n形式・コメントの統一チェックを実施
- ESLint + CI により命名違反をブロック
- フォルダ構成変更時は `import` 修正を自動スクリプトで対応推奨
