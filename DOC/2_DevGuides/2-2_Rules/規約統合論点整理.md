# 規約統合論点整理

## 1. 目的

本書は `規約改善実行計画.md` に基づき、規約統合作業を進めるうえで意思決定が必要な論点を整理するための文書である。

本書で論点を確定した後、個別規約の統合作業を進める。

---

## 2. 論点一覧

| 論点ID | テーマ | 決めること | 影響範囲 |
|--------|--------|------------|----------|
| D01 | ブランチ戦略 | どのブランチ体系を標準運用とするか | Git 運用、CI/CD、レビュー、保護設定 |
| D02 | バックエンド配置 | 新規 API の配置方針を現行追認型にするか、ドメイン分割型にするか | API 実装、レビュー、ファイル配置 |
| D03 | フロントエンド命名 | FE 実装に合わせて共通命名規則を修正するか、共通命名規則に FE を寄せるか | FE 命名、構成、Lint、改名作業 |

---

## 3. 論点詳細

### D01. ブランチ戦略の標準をどちらにするか

#### 現状

- 社内向け簡略版: `main / develop / release / splint / feature / bugfix / hotfix`
- 拡張版: `develop-LD / release-develop / release-uat / release-prd` を含む
- Subtree 連携は別文書で `develop-crj / subtree-merge` を扱っている

#### 選択肢

| 選択肢 | 内容 | メリット | 懸念 |
|--------|------|----------|------|
| D01-A | 社内簡略版を正本にし、拡張版は案件別補足にする | 読みやすく、通常運用の判断が早い | ベンダー案件では補足参照が必須になる |
| D01-B | 拡張版を正本にし、社内簡略版を省略版扱いにする | 1本で多くのケースを説明できる | 通常運用には複雑で、不要分岐が増える |

#### 決定

`D01-A`

決定理由:

- 文書名から見ても `標準ブランチ戦略-社内運用.md` は通常運用の標準に向いている
- `brach4splint.md` はベンダー・複数 release 系の特殊条件を含み、標準運用としては重い
- Subtree 運用は別系統の例外運用として切り出した方が事故を防ぎやすい
- 各 PJ では `release-*` 系拡張版を使用する運用を補足として明記する

---

### D02. バックエンドの新規 API 配置をどちらにするか

#### 現状

- 実コードは `appserver.controller` 直下集約と `request.<category>` ベースが多い
- Repository は `appserver.repository` ではなく `servercommon.repository` に存在する
- もう一方の文書では `reservation` などドメイン単位分割を前提にしている

#### 選択肢

| 選択肢 | 内容 | メリット | 懸念 |
|--------|------|----------|------|
| D02-A | 現行追認型を正本にする | 現行コードと整合し、即運用できる | 将来的なドメイン分割を別途計画する必要がある |
| D02-B | ドメイン分割型を正本にする | 将来の見通しがよく、機能単位で整理しやすい | 現行コードとの乖離が大きく、段階移行計画が必要 |

#### 決定

`D02-A`

決定理由:

- 現時点での規約は、まず誤配置を防ぐことが優先であり、現行実装と整合することが重要
- 実装と離れた理想構成を先に正本化すると、規約違反が大量発生しやすい
- ドメイン分割型は「将来目標」として別章または移行計画へ切り出す方が安全

---

### D03. フロントエンド命名規則をどちらに寄せるか

#### 現状

- FE 実装は `components/base`, `components/composite`, `components/functional` の lower-case グループを採用している
- 共通命名規則はディレクトリ名 `PascalCase` を提示している
- FE 規約の構成図にも `message` ディレクトリが欠けるなど、共通規約とのずれがある

#### 選択肢

| 選択肢 | 内容 | メリット | 懸念 |
|--------|------|----------|------|
| D03-A | FE 実装に合わせて共通命名規則を修正する | 現行コードと整合し、不要な大規模改名を避けられる | 共通規約側の見直しが必要 |
| D03-B | 共通命名規則に FE を寄せる | 規約は単純化しやすい | ディレクトリ改名、import 修正、Lint 調整の影響が大きい |

#### 決定

`D03-A`

決定理由:

- 現行 FE 実装はすでに lower-case グループ構成で安定している
- 命名規則のために大規模なリネームを行うコストが高い
- 共通命名規則を「共通原則」と「FE例外」に整理した方が現実的

---

## 4. 追加で必要になる決定

以下は、上記3論点の決定後に詰める項目である。

- 各規約オーナーのバイネーム
- 例外承認の記録先
- ブランチ保護対象の正式ブランチ一覧
- FE/BE レビュー観点のテンプレート化範囲

---

## 5. 確定結論

以下を確定結論とする。

1. `D01-A`: 社内簡略版を標準運用とする
2. `D02-A`: バックエンド規約は現行追認型を正本化する
3. `D03-A`: FE 実装に合わせて共通命名規則を修正する

上記確定に基づき、次の改訂を進める。

- ブランチ戦略文書の統合
- バックエンド配置規約の一本化
- 命名規則とフロントエンド規約の整合化
