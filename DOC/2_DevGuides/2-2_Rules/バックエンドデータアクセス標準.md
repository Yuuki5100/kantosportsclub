# JPA / MyBatis 技術選定・開発標準

> 文書ステータス: 正本
> 位置づけ: バックエンドのデータアクセスおよびトランザクション設計の基準
> 参照元: `開発ルール総則.md`

## 0. 目的

本ドキュメントは、Spring Data JPA を基本としつつ MyBatis を併用する構成における技術選定基準、設計ルール、実装指針、例外運用、テスト方針を定義する。

本書は、現行正本としてそのまま新規実装と既存改修の両方に適用する。

---

## 1. 本書での読み方

- `必須`: 例外申請がない限り守る
- `推奨`: 標準の実装方法として採用する
- `任意`: チーム判断で採用できる
- `禁止`: 新規実装では採用しない

規約から外れる場合は、`開発ルール総則.md` と `規約例外申請テンプレート.md` に従い、`DOC` 修正と PR/MR 本文記載の両方を行うこと。

---

## 2. 技術選定方針

- 新規の永続化処理は JPA 主体を必須とする
- 以下の条件に当てはまる参照系は MyBatis 利用を推奨する
  - 複雑な検索条件
  - 3 テーブル超の結合
  - 集計、UNION、WITH 句など JPQL で可読性を保ちにくい構造
  - 履歴テーブルや状態遷移の明示的管理が必要なケース
- 更新・削除は JPA を標準とする
- MyBatis による更新・削除は例外申請がある場合のみ許可する

---

## 3. トランザクション制御ルール

### 3.1 新規実装の基準

- `@Transactional` は Service 層にのみ付与することを必須とする
- Controller への `@Transactional` 付与は禁止する
- Repository への `@Transactional` 付与は禁止する
- 読み取り専用処理では `@Transactional(readOnly = true)` を推奨する
- チェック例外でロールバックが必要な場合は `rollbackFor` の明示を推奨する
- `Propagation.REQUIRES_NEW` は禁止する

### 3.2 `REQUIRES_NEW` の例外

以下に限り、例外申請のうえで許可する。

- 通知ログ、エラーログ、監査ログなどの分離副作用処理
- 業務要件上、トランザクション分離が明確に必要であり、影響範囲が文書化されている処理

### 3.3 既存改修時の扱い

- 既存改修でもトランザクション境界は Service 層へ寄せることを必須とする
- 更新・削除の呼び出し元が Service でない場合は、Service を追加してそこへ集約することを推奨する
- トランザクション境界の整理が入る改修では、Repository ではなく Service 側へ寄せる

---

## 4. JPA 利用ルール

- 関連は `fetch = FetchType.LAZY` を明示することを推奨する
- 双方向リレーションは最小限に抑える
- 循環参照回避のため `@ToString.Exclude`, `@JsonIgnore` を適切に使用する
- `JOIN FETCH`, `EntityGraph`, DTO 投影で N+1 問題を回避する
- Entity と DTO は分離し、Controller で Entity を直接返却しない
- Enum 保存は `@Enumerated(EnumType.STRING)` を必須とする
- `CascadeType.ALL` は禁止し、必要最小限の Cascade に限定する

---

## 5. MyBatis 利用ルール

- Mapper は責務単位で分離する
- SQL は `<where>`, `<if>`, `<foreach>` を用いて可読性を維持する
- 共通 SQL は `<sql id="...">` で定義し、重複を避ける
- 共通 SQL に依存する Mapper はテストで整合性を確認する

MyBatis による更新・削除を許可する条件:

- JPA での実装が著しく複雑になる
- ビジネス上の正当な理由がある
- 例外申請が承認されている

---

## 6. `@Query` 使用ルール

- `@Query` は軽量な SELECT に限定して使用する
- 以下をすべて満たす場合にのみ許可する
  - SELECT 文のみ
  - 単一テーブル操作
  - JOIN を含まない
  - シンプルな JPQL で可読性が保たれる
  - 用途が Javadoc またはコメントで明示されている
- 上記条件を外れるクエリは MyBatis へ委譲する
- 例外的に使用する場合は例外申請を必須とする

---

## 7. DTO ↔ Entity マッピング

- DTO ↔ Entity の変換は Mapper 層へ集約することを推奨する
- MapStruct または手動 Mapper を使用してよい
- 命名は用途が分かる形を必須とする
  - `XxxCreateRequest`
  - `XxxUpdateRequest`
  - `XxxQueryResult`
  - `XxxDetailResponse`

null-safe の基準:

- DTO ← Entity で null を初期値へ寄せる場合は `@Mapping(defaultValue = "...")` を使用してよい
- PATCH 系 DTO → Entity では `nullValuePropertyMappingStrategy = IGNORE` を推奨する

---

## 8. パッケージ構成の基準

現行構成に合わせ、以下を標準とする。

- Repository: `com.example.servercommon.repository`
- MyBatis Mapper Interface: `com.example.servercommon.mapper`
- MyBatis XML: Mapper 責務に対応する resources 配下
- 共通 Query DTO: `com.example.servercommon.dto`
- API 向け Request/Response DTO: `com.example.appserver.request.<category>`, `com.example.appserver.response.<category>`
- 手動 Mapper / MapStruct Mapper: `com.example.servercommon.mapper` または責務に対応する mapper パッケージ

`com.example.domain...` のような別体系を新規標準とはしない。

## 9. テスト方針

| テスト種別 | 対象 | 実行環境 | 使用技術・備考 |
|-----------|------|----------|----------------|
| 単体テスト | Mapper / Service | CI / ローカル | `@DataJpaTest`, `@MybatisTest`, `H2` |
| 統合テスト | 全レイヤー | 開発端末 | 実 DB + `@SpringBootTest` |

- CI では H2 による軽量検証を行う
- 開発端末では本番互換 DB での整合性確認を推奨する
- コードカバレッジ目標は 80% とする
- テストデータは `data.sql` などで再現可能にする

マッピングテストの確認項目:

- null 変換
- Enum 変換
- ネスト構造
- リスト変換
- 日付型変換
- `@Mapping(ignore = true)` の動作

---

## 10. TemplateSchema / GenericValidator

- `TemplateSchema` は `schemaVersion` を持つことを必須とする
- ファイル取り込み時は `templateId` または `schemaVersion` で切り替える
- バージョン差分は `GenericFileImporter` で判定する設計を標準とする

---

## 11. 例外運用

以下は例外申請を必須とする。

- Repository への `@Transactional` 付与
- `REQUIRES_NEW` の利用
- MyBatis による更新・削除
- 条件外の `@Query`

例外申請時に必ず残すもの:

- 修正した `DOC`
- PR/MR 本文の例外申請テンプレート
- 影響範囲
- 代替コントロール

本書の標準系は以下に一本化する。

- 更新系トランザクション境界は Service 層に置く
- Repository は永続化操作に専念し、トランザクション境界を持ち込まない
- MyBatis の更新・削除、条件外 `@Query`、`REQUIRES_NEW` は例外運用として扱う
