# 規約検証マトリクス

## 1. 目的

本書は `2-2_Rules` 配下の主要規約について、どの手段で検証するかを整理するための補足文書である。

規約ごとの「守るべき内容」と「どう確認するか」を対応付け、レビューや CI での運用に利用する。

---

## 2. 検証レベル

| レベル | 意味 |
|--------|------|
| CI | 自動検証で判定する |
| Review | レビューで確認する |
| CI + Review | 自動検証とレビューの両方で確認する |
| 設計判断 | 実装前または例外申請時に判断する |

---

## 3. 検証マトリクス

| 対象規約 | 主なルール | 検証方法 | レベル | 備考 |
|----------|------------|----------|--------|------|
| `標準ブランチ戦略-社内運用.md` | `main` / `develop` / `release` への直接 push 禁止 | ブランチ保護設定、PR/MR 運用確認 | CI + Review | 正式な保護対象ブランチ名は `main`, `develop`, `release` に固定する |
| `標準ブランチ戦略-社内運用.md` | 標準フロー `feature -> splint -> develop -> release -> main` | PR/MR のマージ元・マージ先確認 | Review | 案件別拡張は `brach4splint.md` を併用 |
| `ブランチ保護・CI運用一覧.md` | 保護対象ブランチと CI 実行内容の対応 | Git ホスティング設定確認、CI 定義確認 | CI + Review | 標準対象は `main`, `develop`, `release` のみ |
| `brach4splint.md` | `release-*` 系拡張運用 | 案件別のブランチ設計レビュー | 設計判断 | 各 PJ 適用時のみ |
| `バックエンドコーディング規約.md` | API の配置、責務分離、JPA Entity / 共通モデルの `servercommon.model` 集約 | コードレビュー、パッケージ構成確認 | Review | 現行追認型を正本とする |
| `バックエンドコーディング規約.md` | Controller に複雑な業務ロジックを置かない | レビュー、Service 移譲確認 | Review | 静的検知は未整備 |
| `バックエンドデータアクセス標準.md` | `@Transactional` の付与方針 | レビュー、アノテーション確認 | Review | 標準は Service のみ。Repository への付与は例外申請必須 |
| `バックエンドデータアクセス標準.md` | JPA / MyBatis の使い分け | 設計レビュー、実装レビュー | 設計判断 | 複雑 SQL は事前判断が必要 |
| `フロントエンドコーディング規約.md` | MessageCatalog 利用 | ESLint、レビュー | CI + Review | 既存 ESLint ルールあり |
| `フロントエンドコーディング規約.md` | 生文字列エラーの禁止 | ESLint | CI | `eslint.config.mjs` の制約を使用 |
| `フロントエンドコーディング規約.md` | テストカバレッジ 80% 目標 | テスト実行結果、MR 記載確認 | CI + Review | 判定は全体カバレッジのみを基準とする |
| `命名規則.md` | FE の `lang` / Hook / ディレクトリ命名 | ESLint、レビュー | CI + Review | `camelCase + .lang.ts`、Hook は export 名一致を基準とする |
| `規約レビュー運用.md` | MR/PR 記載要件、レビュー観点、承認フロー | テンプレート確認、レビュー履歴確認 | Review | 例外申請時は `規約例外申請テンプレート.md` を併用する |
| `開発ルール総則.md` | 例外申請時の記録必須 | PR/MR 本文、更新した DOC 確認 | Review | DOC 修正と PR/MR 記載の両方を必須とする |

## 4. 現時点で自動化済みの主な項目

- フロントエンドの MessageCatalog 利用
- 生文字列のエラーメッセージ禁止
- Lint 実行
- FE/BE のテスト実行
