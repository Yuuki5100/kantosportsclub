# 開発ルール総則

## 1. 目的

本書は `DOC/2_DevGuides/2-2_Rules` 配下の開発ルール文書について、適用範囲、優先順位、文書ステータス、例外運用の扱いを定義する親文書である。

各規約を個別に読む前に、まず本書を参照すること。

---

## 2. 適用範囲

本書は以下を対象とする。

- DOC 配下の文書配置・仕分け
- Git ブランチ運用
- バックエンド実装規約
- バックエンドデータアクセス標準
- フロントエンド実装規約
- 命名規則
- 規約レビューおよび例外運用

---

## 3. 文書の優先順位

規約の解釈が衝突した場合、以下の順で優先する。

1. 本書 `開発ルール総則.md`
2. テーマ別の正本文書
3. テーマ別の暫定正本文書
4. 補足文書
5. 実装例、サンプル、過去運用資料

補足:

- 共通規約と個別規約が衝突する場合は、個別規約を優先する
- 文書と現行実装が衝突し、かつ正本が未整理の領域では、規約オーナー判断を仰ぐ

---

## 4. ルール強度の定義

各文書では、以下の語をこの意味で使用する。

| 用語 | 意味 | 違反時の扱い |
|------|------|--------------|
| 必須 | 例外規程がない限り守る必要がある | 修正または正式な例外承認が必要 |
| 推奨 | 原則として従うべき標準 | 逸脱理由をレビューで説明する |
| 任意 | 採用可否をチーム判断に委ねる | レビューで整合性確認のみ |
| 禁止 | 原則として認めない | 修正または正式な例外承認が必要 |

以下の曖昧語は、今後の改訂で置換対象とする。

- 原則
- 基本
- なるべく
- できれば
- 必要に応じて

---

## 5. 文書ステータス定義

| ステータス | 意味 |
|-----------|------|
| 正本 | 現時点での正式な基準文書 |
| 暫定正本 | 正本候補だが、統合または改訂前提で当面参照する文書 |
| 補足 | 特定条件下で参照する補助文書 |
| 廃止予定 | 参照停止に向けて移行先を明示した文書 |
| 改訂対象 | 内容見直し中であり、単独では判断根拠にしない文書 |
| 改善案 | 実装寄せや次版改訂の提案を整理した文書 |
| 実行計画 | 規約自体ではなく、改訂作業を管理する文書 |

## 7. 現行運用ルール

確定済みの方針に基づき、以下の扱いとする。

### 7.1 ブランチ戦略

- 社内開発の標準フローは `標準ブランチ戦略-社内運用.md` を正本とする
- 正式な保護対象ブランチ名は `main`, `develop`, `release` に固定する
- 各 PJ で `release-develop`, `release-uat`, `release-prd`, `develop-LD` を使う場合は `brach4splint.md` を補足として併用する
- ブランチ保護と CI 対応の実務参照先は `ブランチ保護・CI運用一覧.md` とする
- Subtree 取り込みは `crj-project` 連携時のみ `サブツリー運用のブランチ戦略追加.md` を参照する

### 7.2 バックエンド配置

- 新規 API の基本配置は `バックエンドコーディング規約.md` を正本とする
- `バックエンドファイル作成規約.md` は廃止予定とし、単独で配置判断に使用しない
- データアクセスに関する判断は `バックエンドデータアクセス標準.md` を優先する
- JPA Entity / 共通モデルの標準配置は `com.example.servercommon.model` とする
- 更新系トランザクション境界は Service 層に統一する

### 7.3 フロントエンド命名

- フロントエンドのファイル/構成/メッセージ運用は `フロントエンドコーディング規約.md` を優先する
- `命名規則.md` は FE の現行実装に合わせて補助指針を更新し、個別規約と矛盾しない内容に保つ
- `lang` ファイルは `camelCase + .lang.ts` に統一する
- 言語状態の参照・更新は `useCurrentLanguage`、辞書選択は `useLanguage` を標準とする
- Hook ファイル名は export 名と一致させる

### 7.4 差分・レビュー・例外・文書管理

- 規約と現行実装の差分判断は `規約差分一覧.md` を参照する
- MR/PR の記載要件とレビュー観点は `規約レビュー運用.md` を参照する
- 文書オーナー、承認者、問い合わせ先の整理は `文書オーナー一覧.md` を参照する
- 廃止予定文書と移行先は `廃止・移行一覧.md` を参照する
- `DOC` 配下の配置判断は `ドキュメント配置・仕分けルール.md` を正本とする
- 派生PJからの移行資料は、正本へ直接混在させず差分・調査資料として分離する

## 8. 例外運用

規約から外れる実装を行う場合は、以下を記録すること。

- 対象規約
- 逸脱内容
- 逸脱理由
- 影響範囲
- 解消予定の有無
- 承認者

例外は口頭合意で完結させず、`DOC` 配下の関連文書修正と PR/MR 本文の両方に残すこと。

PR/MR では、修正した `DOC` を変更対象に含めたうえで、`規約例外申請テンプレート.md` に沿って承認依頼を記載すること。

## 10. 確定済みの主要方針

以下は、本総則に反映済みの確定方針である。

1. ブランチ戦略は `標準ブランチ戦略-社内運用.md` を正本とし、各 PJ の `release-*` 系拡張運用は `brach4splint.md` を補足として使用する
2. 正式な保護対象ブランチ名は `main`, `develop`, `release` に固定する
3. バックエンド配置は現行追認型を正本とする
4. フロントエンド命名規則は現行 FE 実装に合わせて共通命名規則側を修正する
5. フロントエンドのカバレッジ判定は全体カバレッジのみを基準とする
6. JPA Entity / 共通モデルは `com.example.servercommon.model` に統一する
7. 更新系トランザクション境界は Service 層に統一する
8. FE の `lang` / 言語系 Hook / Hook ファイル名は現行実装に合わせて単一命名へ統一する
9. ブランチ命名は `feature/<id>` 形式の slash 区切りを標準とする
10. `バックエンドファイル作成規約.md` は廃止予定文書として扱い、判断根拠を `バックエンドコーディング規約.md` に集約する
11. `DOC` 配下の文書配置判断は `ドキュメント配置・仕分けルール.md` に統一する

---

## 11. 更新ルール

- 新規規約文書を追加する場合は、本書の「現時点の文書位置づけ」を更新する
- 既存文書のステータスを変更する場合は、本書を先に更新する
- 正本の変更は、関連文書のリンクと参照関係を同時に更新する
