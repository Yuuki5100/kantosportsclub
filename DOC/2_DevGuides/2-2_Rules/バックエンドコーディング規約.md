# バックエンド コーディング規約

> 文書ステータス: 正本
> 位置づけ: 現行追認型のバックエンド配置・責務・実装方針の基準
> 参照元: `開発ルール総則.md`

## 0. 目的

本規約は、現行のバックエンド構成（multi-module: `gateway / appserver / batchserver / servercommon / syncconnector`）および、現状の実装スタイル（例: `AdminController`、`JobRunnerController`、`AuthorizationInterceptor`、DTO/Entity 配置）に合わせて、新規 API・機能追加時の置き場所と書き方を統一することを目的とする。

本書は現行実装を追認する形で正本化する。ドメイン単位の全面再編は将来の拡張案として扱い、本書では現時点での誤配置防止を優先する。

---

## 1. 本書での読み方

- `必須`: 例外申請がない限り守る
- `推奨`: 標準の実装方法として採用する
- `任意`: チーム判断で採用できる
- `禁止`: 新規実装では採用しない

規約から外れる場合は、`開発ルール総則.md` と `規約例外申請テンプレート.md` に従い、`DOC` 修正と PR/MR 本文記載の両方を行うこと。

---

## 2. モジュール別の責務

### 2.1 gateway

- `gateway` は認証・認可の入口、ルーティング、フィルタ処理を担う
- 業務ロジックおよび業務 API の実装は禁止
- 許可する実装は、ルーティング設定、フィルタ、入口例外処理に限定する

### 2.2 appserver

- 新規の業務 API は `appserver` に実装することを必須とする
- `appserver` は Controller、Service、Security、Interceptor などの業務 API 本体を担う

### 2.3 batchserver

- `batchserver` はバッチ処理、Job、Tasklet、Scheduler を担う
- 業務 API の実装は禁止とする
- 運用・管理用途の最小限の API のみ、例外申請のうえで許可する

### 2.4 servercommon

- `servercommon` は複数モジュールで共通利用する部品を配置する
- 配置対象は DTO、Model、Repository、共通 Service、Utility、Template とする
- 依存方向は `appserver / batchserver / syncconnector -> servercommon` を必須とし、逆依存は禁止する

### 2.5 syncconnector

- `syncconnector` は外部連携専用モジュールとする
- 外部仕様に依存する HTTP Client、Interceptor、Signature、外部 DTO は `syncconnector` に隔離する

## 3. パッケージ配置規約

### 3.1 appserver の配置基準

以下を標準とする。

- Controller: `com.example.appserver.controller`
- Service: `com.example.appserver.service`
- Interceptor: `com.example.appserver.interceptor`
- Security: `com.example.appserver.security`
- Request DTO: `com.example.appserver.request.<category>`
- Response DTO: `com.example.appserver.response.<category>`

補足:

- 現行実装では `request.admin`, `request.auth`, `request.user` のようなカテゴリ単位を採用している
- 新規実装でもこのカテゴリ単位を優先する

### 3.2 servercommon の配置基準

以下を標準とする。

- Repository: `com.example.servercommon.repository`
- 共通 DTO: `com.example.servercommon.dto`
- 共通 Service: `com.example.servercommon.service`
- 共通 Response モデル: `com.example.servercommon.responseModel`
- JPA Entity / 共通モデルは `com.example.servercommon.model` に統一する

### 3.3 Repository 配置の基準

- 新規 Repository は `com.example.servercommon.repository` に配置することを必須とする
- `com.example.appserver.repository` は現行正本の配置先ではないため、新設しない

---

## 4. 新規 API 追加時の置き場所ルール

### 4.1 Controller の置き方

- 新規 Controller は `com.example.appserver.controller` 配下に追加することを必須とする
- Controller が増えて見通しが悪くなる場合は、サブパッケージ化を任意で許可する
- サブパッケージ化する場合も、`RequestMapping` のルート単位でまとめることを推奨する

例:

- `/admin/**` は `Admin*Controller`
- `/batch-runner/**` は `JobRunner*Controller`

### 4.2 Request / Response DTO の置き方

- 当該 API のみで使用する DTO は `appserver.request.<category>` / `appserver.response.<category>` に置くことを必須とする
- 複数モジュールで共有する DTO は `servercommon.dto` または `servercommon.responseModel` に置く
- 外部連携専用 DTO は `syncconnector.model.<feature>` に置くことを必須とする

### 4.3 servercommon に置いてよい条件

以下のいずれかを満たす場合のみ `servercommon` に配置してよい。

- `appserver` と `batchserver` または `syncconnector` で共通利用する
- 永続化 Model として共通管理する必要がある
- 共通 Repository / 共通 Service と一体で扱う必要がある

迷った場合の基準:

- DTO はまず `appserver` に置く
- Repository は `servercommon.repository` に置く
- Model は `servercommon.model` に置く

## 5. Controller 実装規約

### 5.1 DI / アノテーション

- 依存性注入はコンストラクタ注入を必須とする
- Lombok を使用する場合は `@RequiredArgsConstructor` を推奨する
- 注入対象フィールドは `final` を推奨する

### 5.2 レスポンス / 例外

- API レスポンスは `ApiResponse<T>` を標準とする
- HTTP ステータスコードは `ResponseEntity` で明示する
- エラーメッセージは `ErrorCodeService` 等の共通機構から取得することを推奨する

### 5.3 禁止事項

- `System.out.println` の使用は禁止
- Controller への複雑な業務ロジックの記述は禁止
- 業務判断、永続化制御、外部連携制御は Service に委譲する

### 5.4 Optional の扱い

- `findById` などの取得結果は `Optional` で扱うことを推奨する
- 存在チェックは `orElseThrow` を推奨する
- 既存実装との整合上、`isEmpty()` / `isPresent()` による分岐も許容する

---

## 6. 権限制御 / Interceptor 規約

### 6.1 権限判定の責務

- Controller では `@PreAuthorize(...)` による宣言的制御を標準とする
- ルーティング全体での制御が必要な場合は `AuthorizationInterceptor` を使用する

### 6.2 Interceptor の置き方

- Interceptor は `com.example.appserver.interceptor` に配置することを必須とする
- 認可判定の中心は `PermissionChecker` に寄せる
- Interceptor は前処理に徹し、業務ロジックを持ち込まない
- 403 返却は `HttpServletResponse.SC_FORBIDDEN` を使用し、ログを残す

---

## 7. バッチ起動系 API の取り扱い

- バッチ起動の HTTP 入口は `appserver` に置くことを必須とする
- エンドポイントは `@RequestMapping("/batch-runner")` のように用途が分かるルートへまとめる
- パラメータ妥当性は Controller で最低限チェックし、処理本体は Service に委譲する

---

## 8. servercommon の DTO / Model 規約

### 8.1 DTO

- DB 読み取り結果など、複数箇所で使う DTO は `servercommon.dto` に置く
- DTO は簡潔な表現を推奨する
- 可変 DTO は `@Data`、不変 DTO は `@Value` を使い分けてよい

### 8.2 Model

- JPA Entity には `@Entity` / `@Table` を明示する
- カラム制約は `nullable`, `length` などを明示することを推奨する
- `@Transient` は DB 非保持の拡張用途に限定し、理由コメントを残す

---

## 9. 命名規約

- Controller: `XxxController`
- Service: `XxxService`, `XxxServiceImpl`
- Request DTO: `XxxRequest`
- Response DTO: `XxxResponse`
- Interceptor: `XxxInterceptor`

---

## 10. 新規 API 追加時の最小セット例

例: `/admin/user/permissions` に類する管理系 API を追加する場合

- Controller: `com.example.appserver.controller.AdminController` へ集約、または `controller.admin.*` を新設
- Request: `com.example.appserver.request.admin.*`
- Response: 基本は `ApiResponse<T>` を利用し、必要に応じて `appserver.response.admin.*` を追加
- Repository: `com.example.servercommon.repository`
- Model: `com.example.servercommon.model`
- 権限: `@PreAuthorize(...)` または `AuthorizationInterceptor` 経由で `PermissionChecker` に委譲

## 11. 移行方針

- 既存実装は無理に移動しない
- 新規追加で肥大化が見えた領域は、サブパッケージ化で吸収する
- ドメイン単位の全面再編は、本書ではなく将来案として別計画で扱う
