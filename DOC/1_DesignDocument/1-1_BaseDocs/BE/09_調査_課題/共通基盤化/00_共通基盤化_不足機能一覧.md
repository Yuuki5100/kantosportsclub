# 共通基盤化_不足機能一覧

## 1. 目的

現行バックエンド実装を WEB システム共通基盤として再利用するにあたり、不足している実装と優先度を整理する。

## 2. 判定前提

- 起点資料:
  - `BE/reuse/*`
  - `03_共通部品/*`
  - `06_外部連携/sync-connector.md`
  - `09_調査_課題/*`
- 実装確認対象:
  - `appserver`
  - `batchserver`
  - `gateway`
  - `servercommon`
  - `syncconnector`（任意導入モジュール）
- 判定方針:
  - 単一案件向けの暫定実装ではなく、複数システムで再利用できること
  - 運用・監査・設定変更・障害復旧を考慮した基盤であること

## 3. 優先度一覧

| 優先度 | 機能名 | 機能概要 | 主な不足点 |
| --- | --- | --- | --- |
| P1 | Sync 送信基盤の実運用化 | HMAC 署名付き送信を業務利用可能な共通連携基盤にする | Outbox、送信履歴、再送、業務用送信 UseCase が未実装 |
| P1 | 非同期ジョブ状態永続化基盤 | 帳票・ダウンロード準備・非同期処理の状態を再起動後も参照可能にする | メモリキャッシュ依存、生成物の永続保管なし |
| P2 | 通知再送制御基盤 | `notify_queue` 通知を運用可能な再送制御付き配信基盤にする | 上限回数、バックオフ、異常退避、運用メトリクス不足 |
| P2 | 認可基盤統合・設定外出し | エンドポイント権限判定を共通化し、案件依存コードを減らす | 判定経路の並立、設定正本の不明確さ、動的更新不足 |
| P3 | 動的設定解決基盤 | DB 管理のシステム設定を共通の Resolver 経由で参照可能にする | CRUD はあるが、共通取得 API、型変換、キャッシュ、履歴利用不足 |

## 4. 機能詳細

### 4-1. P1: Sync 送信基盤の実運用化

- 目的:
  - `syncconnector` をサンプル用途から実運用用途へ引き上げる
  - `sync.outbox.use=true/false` で導入有無を切り替え可能にする
- 不足している実装:
  - `sync_outbox_log` の DDL と永続化
  - 送信予約、送信実行、再送管理サービス
  - `SignedRestTemplate` を使う業務送信入口（有効時）
  - 送信結果の監査、冪等性、失敗再実行

### 4-2. P1: 非同期ジョブ状態永続化基盤

- 目的:
  - 非同期ジョブ結果をプロセス外に保持し、運用時の追跡性を確保する
- 不足している実装:
  - `JobStatusCache` / `JobStatusCacheWithStream` 代替
  - 生成ファイルの永続ストレージ保存
  - ジョブ状態取得 API の永続ストア参照化
  - ジョブ期限切れクリーンアップ

### 4-3. P2: 通知再送制御基盤

- 目的:
  - WebSocket 通知を失敗に強い基盤へ引き上げる
- 不足している実装:
  - 再送上限
  - バックオフ
  - 配信失敗状態の明示
  - dead-letter 相当の退避
  - 運用メトリクス

### 4-4. P2: 認可基盤統合・設定外出し

- 目的:
  - 権限判定の実装方式を揃え、案件追加時の個別修正を減らす
- 不足している実装:
  - 権限設定の正本一本化
  - ロールレベル・要求レベルの外部化
  - リロード機構
  - 判定方式の標準化

### 4-5. P3: 動的設定解決基盤

- 目的:
  - 業務設定を API 更新だけでなく共通参照部品として利用可能にする
- 不足している実装:
  - `SystemSettingResolver` 相当の共通取得部品
  - 型変換
  - キャッシュ
  - 変更履歴参照
  - モジュール横断利用のためのインタフェース

## 5. 実施順序

1. `P1` の 2 件を先行する
2. その上で通知と認可の基盤整理を進める
3. 最後に動的設定解決基盤で運用変更容易性を高める
