# 通知再送制御基盤_仕様書

## 1. 目的

`notify_queue` を利用した通知処理に再送制御と障害管理を追加し、運用可能な通知基盤を提供する。

## 2. 適用範囲

- `NotifyQueueScanService`
- `NotifyQueueRepository`
- `WebSocketNotificationService`

## 3. 構成要素

| 区分 | 名称 | 役割 |
| --- | --- | --- |
| Model | `NotifyQueue` | 通知キュー |
| Service | `NotifyQueueScanService` | 配信ワーカー |
| Service | `WebSocketNotificationService` | 通知実送信 |
| Optional API | `NotifyQueueAdminController` | 手動再送、失敗参照 |

## 4. 追加項目

- `status`
- `nextAttemptAt`
- `lastErrorMessage`
- `maxRetry`

## 5. 状態仕様

| 状態 | 説明 |
| --- | --- |
| `PENDING` | 送信待ち |
| `RETRY_WAIT` | 再送待ち |
| `SENT` | 送信済み |
| `FAILED` | 再送上限到達 |

## 6. 再送仕様

- 送信失敗時は `retryCount` を増加する
- 再送間隔は設定値から決定する
- `retryCount >= maxRetry` で `FAILED`

## 7. eventType 運用

- eventType は定義済み一覧から使用する
- Topic 名は `eventType` から一意に導出する
- 小文字化規則を共通化する

## 8. 処理フロー

1. 送信対象レコードを取得する
2. 送信成功なら `SENT`
3. 送信失敗なら `RETRY_WAIT` または `FAILED`
4. 次回送信時刻を更新する

## 9. 運用仕様

- 失敗件数を定期監視対象にする
- 手動再送を可能にする
- 異常 eventType は警告ログを出す

## 10. テスト観点

- 成功
- 一時失敗
- 上限到達
- eventType 別 Topic 生成
