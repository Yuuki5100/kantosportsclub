# 認可方式共通ルール

## 1. 目的
認可判定の正本・標準経路・互換経路を統一し、機能ごとの実装ぶれを防ぐ。

## 2. 適用範囲
- `@RequirePermission`
- `RequirePermissionInterceptor`
- `RolePermissionChecker`
- `PermissionChecker`
- `CustomPermissionEvaluator`

## 3. 正本ルール
- エンドポイント要求権限の正本は `endpoint_authority_mapping`
- ロール保有権限の正本は `role_permission`
- 判定の最終基準は `statusLevelId` 比較とする

## 4. 経路ルール

### 4-1. 標準経路
- Controller保護対象には必ず `@RequirePermission` を付与する
- `RequirePermissionInterceptor` で注釈未設定を検知した場合は拒否する

### 4-2. 補助経路
- Method Security は補助用途として許容する
- `CustomPermissionEvaluator` は `PermissionChecker` に委譲し、設定で未定義挙動を切替える

### 4-3. 互換経路
- `AuthorizationInterceptor` は `security.authorization.legacy-interceptor-enabled=true` の場合のみ使用
- `EndpointPermissionConfig` は `security.permission.legacy-endpoint-config-enabled=true` の場合のみ使用

## 5. エンドポイントキー規約
- 形式は `"METHOD /path"`
- METHODは大文字
- ワイルドカードは `AntPathMatcher` 互換で定義する

## 6. 拒否/許可の原則
- 標準経路は fail closed（要求未定義・不足権限は拒否）
- Method Security補助経路は `security.permission.method-security-allow-if-unmapped` に従う

## 7. ログ・例外
- 拒否時文言は `BackendMessageCatalog.MSG_ACCESS_DENIED` を利用
- 注釈不足は `BackendMessageCatalog.EX_REQUIRE_PERMISSION_MISSING` を利用
- 拒否ログは `LOG_REQUIRE_PERMISSION_ACCESS_DENIED` を利用

## 8. NGパターン
- `roleId` / `statusLevelId` をコードへ固定値直書きする
- 権限定義をDB正本以外へ重複保持する
- 旧経路を設定なしで常時有効にする
