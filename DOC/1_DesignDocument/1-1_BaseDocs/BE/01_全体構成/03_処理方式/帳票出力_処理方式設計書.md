# 帳票出力_処理方式設計書

## 目的
帳票出力の同期/非同期処理方式と、完了通知（notify_queue）の処理方式を実装に合わせて定義する。

## 1. 同期処理（即時生成）
- `POST /report/export/{type}/file`: Base64返却
- `POST /report/download/{type}`: バイナリ返却
- 実体: `ReportService` による即時生成

## 2. 非同期処理（ジョブ + ポーリング）

### 2-1. 起動
- `POST /report/job`
- `ReportTypeJudge` が `jobName` を発行し、`async_job_execution` に `PENDING` 登録後、`ReportPollingRunner` を起動。

### 2-2. 実行
- `ReportPollingRunner` が `RUNNING` へ更新。
- 成果物生成後、`artifact_path` を保存し `COMPLETED` へ更新。
- 失敗時は `FAILED` と `error_message` を保存。

### 2-3. 状態参照
- `GET /report/polling/{jobName}`
- 主参照: `async_job_execution`
- 互換参照: `job_status`（過去データ/旧利用者向け）

### 2-4. 期限切れ処理
- `AsyncJobCleanupService` が `COMPLETED/FAILED` かつ `expires_at` 到達データを定期走査。
- 成果物削除後、`EXPIRED` へ更新。

## 3. downloadReady/download（既存フロー維持）
- `POST /import/downloadReady`: `jobName` 発行 + 非同期生成開始。
- `POST /import/download`: `async_job_execution` の `COMPLETED` を確認し `artifact_path` から再取得して返却。

## 4. 帳票完了通知（notify_queue再送制御）
- `ReportPollingRunner` は帳票完了時に `NotifyQueuePublisher.publish(NotifyEventType.FILE_DOWNLOAD_COMPLETED, reportId)` を実行。
- 通知キューは `PENDING` で登録される。
- `NotifyQueueScanService` が `PENDING/RETRY_WAIT` かつ `next_attempt_at <= now` を対象に送信。
- 送信成功時は `SENT`、失敗時はバックオフを計算して `RETRY_WAIT`、上限到達で `FAILED`。

## 5. 一括帳票出力
- `POST /report/export/bulk`
- `BulkReportExportServiceImpl` が生成・保存し、URLを返却する。

## 6. 整合ルール
- メモリキャッシュへの依存を禁止し、状態はDBを正とする。
- DB更新とストレージI/Oの整合は「失敗時の成果物削除 + FAILED更新」で担保する。
- 通知再送は `notify_queue.status` と `retry_count/next_attempt_at` で制御し、単純な `notified` 判定のみに依存しない。
