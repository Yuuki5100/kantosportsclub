# 動的設定解決基盤_処理方式設計書

## 1. 目的
動的設定の参照・更新時の処理経路を統一し、キャッシュと履歴の整合を担保する。

## 2. 参照処理

### 2-1. 基本フロー
1. 呼び出し元が `SystemSettingResolver` の型付きAPIを呼び出す。
2. `SystemSettingCache` からキーを検索する。
3. キャッシュヒット時は値を返却する。
4. キャッシュミス時は `SystemSettingRepository.findById("1")` で読み込む。
5. `SystemSettingValueMapper` でキーに対応する項目を抽出する。
6. `SystemSettingValueConverter` で要求型に変換して返却する。

### 2-2. 互換キー
- `NUMBER_OF_NOTICES` を正本キーとする。
- 既存互換として `noticeDisplayLimit` も同値を返却する。

## 3. 更新処理

### 3-1. 基本フロー
1. `SystemSettingService.upsert()` が更新要求を受ける。
2. 更新前スナップショットを保持する。
3. `system_setting` を更新（存在しない場合は新規作成）する。
4. 更新後値と比較し、差分のみ `system_setting_history` に保存する。
5. `SystemSettingResolver.evictAll()` でキャッシュを無効化する。

### 3-2. トランザクション
- `upsert()` をトランザクション境界とし、設定更新と履歴保存を同一Txに含める。
- キャッシュ無効化は更新成功時のみ実施する。

## 4. 履歴保存ルール
- 変更がないキーは履歴保存しない。
- 履歴には `before_value / after_value / updated_by / updated_date_and_time` を記録する。
- 新規作成時は `before_value = null` で登録する。

## 5. 失敗時挙動
- 型変換失敗: `IllegalArgumentException` を送出し、呼び出し元でハンドリングする。
- 設定行未存在: `Optional.empty` を返却し、`getOrDefault` 系で既定値利用を許容する。
- 更新失敗: トランザクションロールバックし、履歴・キャッシュは更新しない。
