# フロントエンド モックモード 基本設計書

## 1. 概要

本ドキュメントは、フロントエンドに内蔵するモックモード（Demo Mode）の基本設計を定義する。
モックモードは、バックエンドやDBに接続せずにUI確認・デモを可能にする機能であり、既存の認証基盤・API共通層・画面構成を維持したまま切替可能とする。

---

## 2. 目的

- バックエンド未接続でも画面確認を可能にする
- デモ用途で一貫した操作体験を提供する
- 既存設計（API層・認証・通知）を崩さずに切替できるようにする

---

## 3. スコープ

### 対象
- 環境変数1つでモックモードON/OFF
- APIレスポンスのモック化
- ログイン/認証のモック化
- 一覧/詳細/更新のUI確認

### 対象外
- 外部モックサーバー
- バックエンド変更
- DB利用

---

## 4. 全体アーキテクチャ

```text
[Page / Component]
        ↓
[useApi / hooks]
        ↓
[services (authService, userService, ...)]
        ↓
[apiService]
        ↓
[apiClient]

※ モックモード時は services 層でモック分岐
```

### 方針
- 画面・コンポーネントではモック分岐しない
- API呼び出しの境界（service層）で切替する

---

## 5. 環境変数設計

```env
NEXT_PUBLIC_USE_MOCK_MODE=true
```

### 要件
- 未設定時は通常モード
- クライアントから参照可能
- 共通ユーティリティ経由で取得

---

## 6. モック切替設計

### 6.1 共通ユーティリティ

```ts
isMockMode(): boolean
```

責務:
- 環境変数を取得
- 全体でモック状態を統一

---

## 7. APIモック設計

### 方針
- service層で分岐
- API呼び出しの代わりにモックデータを返す

```ts
if (isMockMode()) {
  return mockUserList()
}
return apiService.get(...)
```

### モック対象
- 一覧取得
- 詳細取得
- 更新API

---

## 8. 認証モック設計

### 対象
- login
- logout
- auth status

### 挙動

#### login
- 常に成功
- モックユーザー返却

#### auth status
- 常に認証済み

#### logout
- ローカル状態クリア

### モックユーザー

```ts
{
  username: "demo-admin",
  rolePermissions: {
    "/admin": 3
  }
}
```

---

## 9. モックデータ設計

### 配置

```text
src/mocks/
  ├─ auth/
  ├─ user/
  ├─ admin/
```

### 方針
- ドメイン単位で分離
- 固定データ or 簡易生成

---

## 10. 画面影響範囲

### 対象画面
- 一覧画面
- 詳細画面
- フォーム画面

### 非対象
- WebSocket（今回対象外）

---

## 11. 更新処理設計

### 方針
- APIは呼ばない
- UI上は成功扱い

### オプション
- メモリ上で状態更新

---

## 12. エラー再現設計

### 方針
- モックでエラーを返せるようにする

例:
- 一覧失敗
- 権限不足

---

## 13. 非機能要件

### 保守性
- モックは1箇所で制御
- 画面コードに影響を与えない

### 拡張性
- API単位で追加可能

### セキュリティ
- 本番で無効化運用

---

## 14. ファイル構成

```text
src/
  ├─ mocks/
  ├─ services/
  ├─ utils/envUtils.ts
```

---

## 15. 受入条件

- envフラグで切替可能
- ログインできる
- API不要で画面動作
- 既存基盤を壊さない

---

## 16. 要確認事項

- モック対象API範囲
- モックユーザー種別
- 本番での無効化ルール

---

## 17. 実装方針まとめ

- service層でモック分岐
- envフラグで制御
- 認証も同様にモック化
- UIは変更しない
