# フロントエンド モックモード 詳細設計書

## 1. 目的

本ドキュメントは、フロントエンドに内蔵するモックモード（Demo Mode）の詳細設計を定義する。
本機能は、バックエンドやDBに接続せずに、フロントエンドに存在する全API呼び出しをモック化し、UI確認・デモ・画面開発を可能にすることを目的とする。

---

## 2. 前提

- 切替は `.env` のフラグ1つで行う
- 別途モックサーバーは起動しない
- 既存画面コードにモック用の条件分岐を極力持ち込まない
- 既存の認証基盤、API共通層、エラーハンドリング、通知方式を維持する
- 現在フロントエンドに存在する全APIを対象とする

---

## 3. 対象範囲

### 3.1 対象
- 認証API
- 一覧取得API
- 詳細取得API
- 登録API
- 更新API
- 削除API
- マスタ取得API
- 設定値取得API
- その他、既存 `services/` もしくは `useApi` 経由で利用される全API

### 3.2 対象外
- WebSocket通信の完全モック（今回は対象外）
- バックエンド実装変更
- DB接続
- 外部モックサーバー

---

## 4. 全体方針

### 4.1 基本方針

モック切替は **service層で実施** する。

```text
Page / Component
  ↓
Hook / useApi
  ↓
Service層
  ├─ Mock Mode ON  → mock handler を返却
  └─ Mock Mode OFF → apiService / apiClient を呼び出す
```

### 4.2 採用理由

- 画面側の実装を変更しなくてよい
- 既存のAPI責務を壊さない
- API単位で差し替えしやすい
- 認証・権限・通知も既存導線のまま利用できる

---

## 5. 環境変数設計

### 5.1 利用環境変数

```env
NEXT_PUBLIC_USE_MOCK_MODE=true
```

### 5.2 利用ルール

- `true` の場合のみモックモードを有効化
- 未設定または `false` の場合は通常モード
- 取得は共通ユーティリティ経由で行う

### 5.3 共通ユーティリティ

```ts
export const isMockMode = (): boolean => {
  return process.env.NEXT_PUBLIC_USE_MOCK_MODE === 'true'
}
```

配置候補:

```text
src/utils/envUtils.ts
```

## 7. APIモック方式

## 7.1 基本形

各 service で以下のように分岐する。

```ts
export const fetchUserList = async (params: UserListParams) => {
  if (isMockMode()) {
    return mockUserHandlers.fetchUserList(params)
  }
  return apiService.get('/users', { params })
}
```

## 7.2 実装ルール

- 画面側ではモック判定しない
- `services/*` 単位でモックへ切替
- 戻り値の型は通常APIと揃える
- Promise を返す
- 必要に応じて疑似遅延をかけられる構造にする

## 8. 認証モック詳細設計

## 8.1 対象API

- login
- logout
- auth status
- 必要に応じて forgot-password / reset-password

## 8.2 モックユーザー定義

```ts
interface MockUser {
  username: string
  displayName: string
  email: string
  role: string
  rolePermissions: Record<string, number>
}
```

例:

```ts
export const mockUsers = {
  admin: {
    username: 'demo-admin',
    displayName: 'Demo Admin',
    email: 'demo-admin@example.com',
    role: 'ADMIN',
    rolePermissions: {
      '/admin': 3,
      '/user': 3,
      '/role': 3,
      '/settings': 3,
      '/report': 2,
      '/notice': 2,
    },
  },
  user: {
    username: 'demo-user',
    displayName: 'Demo User',
    email: 'demo-user@example.com',
    role: 'USER',
    rolePermissions: {
      '/myPage': 2,
      '/notice': 1,
    },
  },
}
```

## 8.3 login 動作

入力:
- username
- password

動作:
- password の厳密判定は不要
- username でモックユーザーを選択
- 未指定時はデフォルトで `demo-admin`

返却:
- 通常ログインAPIと同等のレスポンス構造

## 8.4 auth status 動作

- モックセッション情報が存在すれば認証済みを返す
- 存在しなければ未認証を返す

## 8.5 logout 動作

- モックセッション情報を破棄する
- 未認証状態へ戻す

## 8.6 セッション管理

選択肢:
- in-memory
- localStorage
- sessionStorage

推奨:
- `sessionStorage`

理由:
- タブ単位で扱いやすい
- デモ用途に十分
- Cookieやサーバーセッションを不要にできる

## 10. モックレスポンス設計

## 10.1 共通レスポンス方針

既存APIレスポンス構造に合わせる。

想定共通形:

```ts
interface MockListResponse<T> {
  list: T[]
  total: number
}
```

```ts
interface MockDetailResponse<T> {
  data: T
}
```

```ts
interface MockMutationResponse {
  success: boolean
  message?: string
}
```

※ 実際は既存APIの返却型に合わせて定義する。

## 10.2 一覧共通処理

共通関数:
- filter
- sort
- paginate

```ts
filterList(data, params)
sortList(data, sortKey, sortOrder)
paginateList(data, page, size)
```

これにより、全一覧APIで同じUI挙動を再現する。

## 11. 画面別モック要件

## 11.1 一覧画面

- 20件以上のダミーデータを持つこと
- ページング確認可能であること
- ソート確認可能であること
- 検索結果0件を再現できること
- 長文・短文・空欄混在を含むこと

## 11.2 詳細画面

- 主な表示項目をすべて埋めたデータを持つこと
- 一部空欄パターンを持つこと
- ステータス違いパターンを持つこと

## 11.3 フォーム画面

- 初期値表示可能であること
- 更新成功を返せること
- バリデーション確認用の異常データを用意可能であること

---

## 12. 更新系モック詳細

## 12.1 方針

- 登録/更新/削除は成功レスポンスを返す
- 必要に応じて sessionStorage 上のモックデータを更新する
- 画面再描画後も整合が取れるようにする

## 12.2 実装段階

### 第1段階
- 成功レスポンスのみ返す
- 一覧再取得時は固定データのまま

### 第2段階（任意）
- 画面中の更新結果をメモリまたは sessionStorage へ反映

推奨:
- まず第1段階を採用

理由:
- UI確認用途としては十分
- 実装コストを抑えられる
- データ整合崩れのリスクが低い

---

## 13. エラー再現設計

## 13.1 共通方針

## 13.2 再現方法

  permissionDenied: false,

## 15. 通知・エラー処理設計

## 15.1 成功通知

- 更新成功
- 削除成功
- 登録成功

## 15.2 異常通知

- 一覧取得失敗
- 認証失敗
- 権限不足

## 15.3 実装方針

- 既存の `errorHandler` を利用する

- 画面個別の独自通知は作らない

## 16. 実装ステップ

### Step 1
- 既存 `services/` のAPI呼び出しを洗い出す
- API一覧表を作成する

### Step 2
- `isMockMode()` を実装する
- `src/mocks/` 基盤を作成する

### Step 3
- auth系 API をモック化する
- login / logout / auth status を切替対応する

### Step 4
- 一覧系 API を順次モック化する
- list/detail/create/update/delete の handler を作る

### Step 5
- エラーシナリオと権限差分を追加する

### Step 6
- 対象画面を動作確認する

## 18. テスト観点

## 18.1 正常系
- env フラグ ON でモックモードへ切替できる
- ログインできる
- 認証済み画面へ遷移できる
- 一覧表示できる
- 詳細表示できる
- 更新成功通知が表示される

## 18.2 異常系
- 一覧取得エラーを再現できる
- 認証失敗を再現できる
- 権限不足を再現できる
- 0件一覧を表示できる

## 18.3 切替系
- env フラグ OFF で実APIモードに戻る
- 画面コードに追加修正なしで切替できる

## 18.4 構造系
- 既存の責務分離を壊していない
- 画面側にモック分岐が入っていない
- 直接 fetch / axios を追加していない

---

## 20. 実装方針まとめ

- モックモードは env フラグ1つで切替する
- service層でモック分岐を行う
- 既存画面コードには原則手を入れない
- 認証も既存 auth 基盤に乗せたままモック化する
- フロントに存在する全APIを順次モック対象とする
- 一覧共通処理・レスポンス整形は mocks/common に集約する
- 通知・エラーは既存基盤を利用する
