# 目的
共通モジュール利用状況の将来的な集計精度を上げるために、以下の変更を今後実施する前提で、現状調査・修正対象整理・実行計画のドキュメント化を行う。

対象テーマ:
- tsconfig の alias 整理
- import rule の統一
- 公開入口（index.ts）の整備方針
- 関連ドキュメント更新
- 変更対象ファイルの明確化
- 実行計画の文書化

# 背景
- components 配下のレイヤ構成は base / composite / functional / CRJ
- 集計対象は functional / composite / CRJ
- base は利用可だが追跡対象外
- import の表記ゆれを減らし、将来的に「どのページでどの共通モジュールが使われているか」を機械集計しやすくしたい
- 今回はまず規約統一の土台を作る
- 既存ドキュメントの更新対象は DOC/1_DesignDocument/1-1_BaseDocs/FE 配下
- composite ディレクトリ名は composite で統一し、composit という表記は使わない

# 今回の意思決定
- components 系 alias は @/components/... に寄せる
- @components/*, @composite/*, @base/*, @functional/* のような components 専用 alias は廃止方向
- 原則として @/* を基準 alias とする
- 追跡対象モジュールは公開入口から import する
- 追跡対象モジュールの default export は増やさず、named export を基本方針とする
- test / stories / sample は集計対象外
- base は追跡対象外なので、現時点では import 制約を強くしすぎない

# 重要制約
- **コード修正禁止**
- **設定ファイル修正禁止**
- **ドキュメント新規作成・更新は可**
- 変更提案はしてよいが、実装は次回に回す
- 調査結果は、後続の実装担当がそのまま使える粒度で残す
- 勝手に import 一括置換や大規模修正を始めない

# 今回やること
1. 現状調査
2. 修正対象ファイル一覧の作成
3. 実行計画ドキュメントの作成
4. 実装時の注意点整理
5. 最終的な実装対象・非対象の明確化

# Step 1: 現状調査
以下を確認してください。
- tsconfig.json の実在位置
- components 関連 alias の現状
- ESLint 設定ファイルの有無（.eslintrc.*, eslint.config.* など）
- src/components/composite/index.ts の有無
- src/components/functional/index.ts の有無
- src/components/CRJ/index.ts の有無
- DOC/1_DesignDocument/1-1_BaseDocs/FE 配下のドキュメント一覧
- import rule / コーディング規約 / コンポーネント設計規約が書かれていそうなドキュメントの特定
- components 専用 alias を実際に使っていそうなファイル群の把握
- deep import が多そうな箇所の傾向把握
- 実装時に影響を受けそうな index.ts / lint / tsconfig 関連ファイルの把握

出力:
- 調査結果サマリ
- 実装時に編集候補となるファイル一覧
- 直接修正しないが影響確認が必要なファイル一覧
- 将来対応候補ファイル一覧
- 実装前に確認すべきリスク一覧

# Step 2: 修正対象ファイル一覧ドキュメントの作成

このドキュメントに含める内容:

- 背景

- 将来対応候補ファイル一覧

実装計画をドキュメント化してください。
