修正済みドキュメント（DOC\1_DesignDocument\1-1_BaseDocs）を基に、
実装判断用の一覧表を作成してください。

# 前提
- ドキュメントは現行実装に合わせて修正済み
- 実装コードの確認は不要
- ドキュメントを唯一の正とする

# 目的
各機能について後続開発フェーズにて、以下を即判断できるようにする
- 何に使うか
- 流用可能か
- 改修が必要か
- 新規実装が必要か

# 出力ファイル（必須）
以下の6ファイルを作成すること

1. BE_reference.md
2. BE_reference.csv
3. FE_reference.md
4. FE_reference.csv
5. integration_reference.md
6. integration_reference.csv

# 区分ルール
- BE：バックエンド処理、バッチ、API、サービス
- FE：画面、UIロジック、hooks、components
- integration：FE↔BE、BE↔外部、非同期連携、バッチ連携

# 各ファイルの内容
すべて同一構造の表を使用すること

列は以下：

区分,モジュール/機能名,概要,使用場面/ユースケース,既存機能,判定,関連ドキュメント,I/O概要,依存先,備考

# 判定ルール
- 流用可
- 改修要
- 新規実装
- 要確認

# 作業手順
1. ドキュメントから機能・モジュール・処理を抽出
2. 区分（FE / BE / integration）に分類
3. 各機能について以下を整理
   - 概要
   - 使用場面
   - I/O概要
   - 依存先
4. 判定を付与
5. 区分ごとにファイルへ分割して出力

# 出力ルール

## MDファイル
Markdown表形式

| 区分 | モジュール/機能名 | 概要 | 使用場面/ユースケース | 既存機能 | 判定 | 関連ドキュメント | I/O概要 | 依存先 | 備考 |

## CSVファイル
ヘッダ行あり

区分,モジュール/機能名,概要,使用場面/ユースケース,既存機能,判定,関連ドキュメント,I/O概要,依存先,備考

# 重要ルール
- 実装コードは参照しない
- ドキュメントの記載のみを根拠とする
- 推測は禁止。不明な場合は「要確認」とする
- 「追記予定」「要検討」は必ず備考に記載
- 同じ機能が複数区分にまたがる場合は、それぞれに記載する

# 出力順
1. BE_reference.md
2. FE_reference.md
3. integration_reference.md
4. BE_reference.csv
5. FE_reference.csv
6. integration_reference.csv

# 最終チェック
- 各ファイルに重複がないこと
- 判定が空欄の行がないこと
- 各区分に適切に分類されていること
