# 前提
- 調査と計画は完了済み
- 修正対象ファイル一覧ドキュメントと実行計画ドキュメントが存在する前提で進める

# 目的
共通モジュール利用状況の将来的な集計精度を上げるため、以下を最小差分で実装する。
- tsconfig の alias 整理
- 公開入口（index.ts）の整備
- import rule / lint rule の最小導入
- FE 関連ドキュメント更新

# 背景
- components 配下のレイヤ構成は base / composite / functional / CRJ
- 集計対象は functional / composite / CRJ
- base は利用可だが追跡対象外
- components 系 alias は @/components/... に寄せる
- @components/*, @composite/*, @base/*, @functional/* のような components 専用 alias は廃止方向
- 原則として @/* を基準 alias とする
- 追跡対象モジュールは公開入口から import する
- named export を基本方針とする
- test / stories / sample は集計対象外
- base は現時点で import 制約を強くしすぎない

# 作業ルール
- まず前回作成された対象一覧ドキュメントと実行計画ドキュメントを確認する
- 変更は最小差分
- 既存コード全体の import 一括置換はしない
- 既存 export を壊さない
- lint はいきなり大量エラーにならないよう warning ベースまたは対象限定で導入する
- ドキュメント更新は既存文体に合わせる
- 不明点は勝手に拡張せず、仮定を明示する
- 実装後は diff と影響確認を必ず残す

# Step 2: tsconfig 修正
次の方針で tsconfig を修正してください。
- @/* は維持
- components 専用 alias は削除候補として扱う
  - @components/*
  - @composite/*
  - @base/*
  - @functional/*
- 他の非 components 系 alias（@hooks/*, @api/* など）は原則維持
- 変更は最小差分
- コメントがある場合は既存スタイルに合わせる

期待する方向:
"paths": {
  "@/*": ["src/*"],
  "@hooks/*": ["src/hooks/*"],
  "@api/*": ["src/api/*"],
  "@assets/*": ["src/assets/*"],
  "@utils/*": ["src/utils/*"],
  "@config/*": ["src/config/*"],
  "@types/*": ["src/types/*"],
  "@slices/*": ["src/slices/*"],
  "@lang/*": ["src/lang/*"]
}

出力:
- 修正前後の要点
- diff
- 残した alias / 削除した alias の理由

# Step 3: 公開入口の整備
次を確認し、必要なら整備してください。
- src/components/composite/index.ts
- src/components/functional/index.ts
- src/components/CRJ/index.ts

ルール:
- composite / functional / CRJ は公開入口を用意する
- 既存 export を壊さない
- CRJ/index.ts を新規作成する場合は、CRJ 直下の公開対象だけを export する
- stories/test/util/internal 実装は export しない
- default export の全面修正はしない
- 今回は運用ルール整備が主目的

出力:
- 追加/変更した index.ts の一覧
- export 方針
- 今回見送った点

# Step 4: import rule / lint rule の整備
ESLint 設定がある場合は、既存設定を確認したうえで最小差分でルール追加してください。

導入したい方向:
1. components 配下は @/components/... を使う
2. pages / app / feature / screen 側からの deep import を禁止する
3. composite / functional / CRJ は公開入口 import を推奨または強制する
4. test / stories は必要に応じて例外扱いにする
5. 相対 import で components を深く辿る書き方を禁止または警告にする

候補:
- no-restricted-imports
- import/no-internal-modules
- overrides を使った stories/test 例外
- warning ベース導入

重要:
- repo 全体が大量エラーになる設定は避ける
- まず warning または対象限定
- 将来の統一に向けた最小導入

出力:
- 採用ルール
- 意図
- diff
- 既存影響の見立て
