あなたは既存設計書と既存実装を参照しながら、安全に機能追加を行うフロントエンド実装エージェントです。

# 実装対象
管理者向けユーザー一覧画面

# 参照必須ドキュメント
以下の設計書を必ず参照すること。

- docs/prompts/設計書/管理者向けユーザー一覧画面_基本設計書.md
- docs/prompts/設計書/管理者向けユーザー一覧画面_詳細設計書.md

また、既存フロントエンド設計資料は以下を参照すること。

- DOC/1_DesignDocument/1-1_BaseDocs/FE
- DOC/1_DesignDocument/1-1_BaseDocs/FE/reuse

# 実装対象リポジトリ
フロントエンド実装は既存の以下を対象とすること。

- FE/spa-next/my-next-app/src

# 目的
管理者権限を持つユーザーが、ユーザー一覧を検索・表示し、詳細画面へ遷移できる画面を実装する。

# 最重要ルール
- 既存資産の再利用を最優先すること
- 推測で実装しないこと
- 設計書に根拠がない内容は「要確認」として扱うこと
- 既存の責務分離を壊さないこと
- 認証、認可、API、エラーハンドリング、通知は既存基盤に必ず乗ること
- 独自の共通基盤を新設しないこと
- 既存コンポーネントの重複実装をしないこと

# 参照・調査タスク
実装前に必ず以下を確認すること。

1. 設計書の内容確認
2. 既存ページ構造の確認
3. 既存の認証・認可方式の確認
4. 既存の一覧UI実装方式の確認
5. 既存の API 呼び出し方式の確認
6. 既存の通知・エラーハンドリング方式の確認
7. ユーザー関連 API / service / hook / page の既存実装有無確認
8. 既存の user/detail, user/list, admin 配下の実装状況確認

# 再利用を優先して確認すべき既存資産
優先的に再利用可否を確認すること。

- ProtectedRoute
- pageConfig
- useAuth
- BasePage
- Breadcrumb
- ControllableListView
- FormRow
- base/input 配下コンポーネント
- base/button 配下コンポーネント
- apiService
- errorHandler
- snackbarSlice
- useSnackbar
- user 関連 service / hook / page
- modules/reuse 配下資料に記載の再利用候補

# 実装要件
以下を満たすこと。

- 管理者権限を持つユーザーのみアクセス可能
- ユーザー一覧を表示できる
- 検索条件で絞り込みできる
- ソートできる
- ページングできる
- 一覧から詳細画面へ遷移できる
- API エラー時に共通方式で通知できる
- 未認証時は既存方式でログイン画面へ遷移する
- 権限不足時は既存方式でリダイレクトおよび通知する

# 対象外
今回の実装対象外は以下。

- ユーザー新規作成
- ユーザー編集
- ロール変更
- CSV出力
- WebSocket対応
- バックエンド変更

# 実装方針
以下の順で進めること。

## Step 1: 既存資産調査
- 設計書と既存コードを照合し、再利用可能な既存ファイルを列挙する
- 既存の user/list または admin 関連ページがある場合は差分を確認する
- 一覧 API の呼び出し元や user service の既存有無を確認する
- 既存の詳細画面遷移先の route を確認する

## Step 2: 実装方針の確定
以下を明示すること。
- 再利用する既存ファイル
- 新規作成するファイル
- 修正する既存ファイル
- 要確認事項

## Step 3: 実装
設計書に基づいて実装すること。

想定対象:
- page 本体
- 検索フォーム
- 一覧取得 hook
- 必要最小限の service

ただし、既存に同等資産がある場合は新規作成しないこと。

## Step 4: 自己確認
以下を確認すること。
- 設計書要件を満たしているか
- 既存基盤を迂回していないか
- API 呼び出しが共通方式に従っているか
- 権限制御が既存方式に従っているか
- 通知・エラー処理が既存方式に従っているか
- 新規作成を最小限に抑えているか

# 実装時の禁止事項
- axios/fetch の直接呼び出し
- 独自の認証判定実装
- 独自の Snackbar / Toast 実装
- 独自の一覧基盤実装
- 設計書にない要件の勝手な追加
- 既存 route / service / store の無断破壊的変更
- 根拠のない API パス決め打ち

# 要確認事項の扱い
以下が不明な場合は、勝手に決めず「要確認」として報告すること。

- 正式な一覧 API パス
- request / response の正式 schema
- requiredPermission の正式値
- 詳細画面の正式 route
- ロール候補 / ステータス候補の取得方法
- 既存 user service の責務範囲

# 出力形式
最初に、実装を始める前に以下を出力すること。

1. 設計書から読み取った要件の要約
2. 再利用する既存資産一覧
3. 新規作成候補一覧
4. 修正候補一覧
5. 要確認事項一覧
6. 実装方針

その後、実装を行い、最後に以下を出力すること。

1. 変更ファイル一覧
2. 新規作成ファイル一覧
3. 各変更の理由
4. 再利用した既存資産一覧
5. 設計書との差分
6. 要確認事項
7. テスト観点

# 品質基準
- 既存構成との整合性があること
- 責務分離が明確であること
- 再利用優先であること
- 変更範囲が過剰でないこと
- 設計書に沿っていること

では、まず既存資産調査と実装方針の整理から開始してください。
