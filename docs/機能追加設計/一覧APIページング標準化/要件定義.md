# 一覧APIページング標準化_要件定義

## 1. 目的
一覧系APIのページング仕様を共通化し、フロント/バックエンド双方で再利用可能な実装パターンを確立する。

## 2. 背景
- 一覧APIごとにクエリ名が不統一（`pageNumber/pagesize`、`pageNo/pageSize` など）。
- 0始まり/1始まりの混在があり、FE側で個別変換ロジックが必要になっている。
- ソート指定（`sortKey/sortOrder`）の有無・形式がAPIごとに異なる。
- 将来的な新規一覧API作成時に、毎回個別仕様の判断が必要になっている。

## 3. 対象範囲

### 3-1. 対象
- `/api/user/list`
- `/api/roles/list`
- `/api/manual/list`
- `/system-transfer/batch-list`

### 3-2. 対象外（本課題では実施しない）
- 一覧以外のAPI（詳細/作成/更新/削除）
- 画面レイアウト変更
- 権限モデル変更
- APIバージョン分離（`/v2` 新設など）

## 4. 現行課題（As-Is）
| API | 現行ページング | 現行ソート | 主な課題 |
|---|---|---|---|
| `/api/user/list` | `pageNumber`, `pagesize` | なし | 他APIと名称不一致 |
| `/api/roles/list` | `pageNumber`, `pagesize` | なし | ソート不可、名称不一致 |
| `/api/manual/list` | `pageNumber`, `pagesize` | なし | 名称不一致 |
| `/system-transfer/batch-list` | `pageNo`, `pageSize` | `sortKey`, `sortOrder` | 起点インデックスの扱い差異 |

## 5. 要件（To-Be）

### FR-01 共通クエリ仕様
対象一覧APIは共通で以下のクエリを受け付ける。
- `pageNumber`: ページ番号（1始まり）
- `pageSize`: 1ページ件数
- `sortKey`: ソート対象キー（任意）
- `sortDir`: `asc` / `desc`（任意、未指定時 `asc`）

### FR-02 既存検索条件の維持
既存の業務検索条件（例: `name`, `roleId`, `titleName`, `base`, `batch`）はそのまま利用可能とする。

### FR-03 バリデーション共通化
ページング共通ルールを適用する。
- `pageNumber >= 1`
- `1 <= pageSize <= 100`（上限は設定可能）
- `sortDir` は `asc` / `desc` のみ許可
- 不正値は `400`（既存エラー形式）で返却

### FR-04 ソート制御の安全化
- APIごとに `sortKey` 許可キーをホワイトリストで定義する。
- 未許可キー指定時は `400` を返す。

### FR-05 一括置換
- 旧クエリ（`page` / `size` / `sortBy` / `pagesize` / `pageNo` / `sortOrder`）は受け付けない。
- 対象一覧APIはリリース時点で統一仕様（`pageNumber` / `pageSize` / `sortKey` / `sortDir`）へ全面置換する。
- 旧クエリ指定時は `400` を返す。

### FR-06 レスポンスのページング情報明示
一覧レスポンスは、既存の `total` に加えて次を返却する。
- `pageNumber`
- `pageSize`
- `total`

注記: 既存レスポンス互換を優先し、`items` への全面置換は本課題の対象外とする。

### FR-07 フロント共通利用
- FEサービス層でページングクエリ組み立てを共通化する。
- 画面側は `ControllableListView` の状態をそのまま共通クエリへ変換し、個別変換ロジックを削減する。

## 6. 非機能要件
- パフォーマンス: 既存比で一覧APIの応答性能を劣化させない。
- 保守性: 新規一覧APIは本仕様の共通DTO/共通変換を使うこと。
- 可観測性: 新クエリの入力不正（`pageNumber/pageSize/sortKey/sortDir`）をログで追跡可能にする。

## 7. 導入要件

### 7-1. 導入方針
1. BE/FEを同一リリースで統一仕様に揃える
2. 旧クエリ受付は実装しない
3. 対象4APIを統一仕様で一括リリースする

### 7-2. 置換ポリシー
- 本基盤に利用者はいない前提のため、段階移行は行わない。
- 仕様差分は実装時点で解消し、旧仕様の併存期間を設けない。

## 8. テスト要件

### 8-1. 正常系
- 新クエリで全対象APIが一覧取得できる。
- `pageNumber/pageSize` に応じて件数と総件数が正しく返る。
- `sortKey/sortDir` が想定どおり適用される。

### 8-2. 異常系
- `pageNumber=0`, `pageSize=0`, `pageSize>100`, 不正 `sortDir` で `400`。
- 許可外 `sortKey` で `400`。
- 旧クエリ（`page/size/sortBy/pagesize/pageNo/sortOrder`）指定で `400`。

### 8-3. 置換確認
- 対象4APIが旧クエリを受け付けないことを確認する。
- FE一覧画面が新クエリのみで正常動作することを確認する。

## 9. 受入基準
- 対象4APIが共通ページング仕様（`pageNumber/pageSize/sortKey/sortDir`）で動作する。
- 既存画面が回帰なく利用できる。
- FEの一覧取得コードでAPI固有ページング変換が減っている。
- 旧クエリ指定時に `400` を返す。
